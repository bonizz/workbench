# Milestone 0.11 — Translate Gizmo (Minimal)

## Goal
Drag a selected `MeshRenderer` object across the XZ ground plane (Y preserved)
by clicking a small yellow handle in the viewport. No axis arrows, no rotate/
scale, no undo, no serialization changes.

## What already exists (reuse, don't rebuild)
- `core/picking.{h,cpp}`: `Ray`, `makeCameraRay`, `intersectRaySphere`,
  `intersectRayAABB`, `intersectRayPlane` (the last is **bounded** to the unit
  plane primitive — not reusable for an arbitrary-height drag plane).
- `Application::onMouseButton(0, true, x, y)` already builds a camera ray from
  pixel coords (NDC → `makeCameraRay`) and calls `Scene::pickObject` +
  `editor_->setSelected`. (`core/application.cpp:376`)
- `Editor` owns selection: `selected()` / `setSelected()`.
- `Scene::buildRenderCommands(ctx, selected)` already emits a selection
  highlight overlay for the selected object — the natural home for the gizmo
  visual. (`scene/scene.cpp:144`)
- `transform().position` is a public `Vec3` field (local). For an unparented
  object local == world. Inspector/commands edit local; we do the same.
- Platform: `mouseDown:` is wired; **`mouseDragged:`/`mouseUp:` are not** — left
  drag currently does nothing (only right-drag orbits the camera).

## Smallest implementation

### 1. New pure math helpers — `core/picking.{h,cpp}` (testable, deterministic)
```cpp
// Unbounded horizontal plane y = planeY. Returns false if ray is parallel or
// the hit is behind the origin. On hit, fills hitPoint.
bool intersectRayHorizontalPlane(const Vec3& origin, const Vec3& dir,
                                 float planeY, Vec3& hitPoint);

// Drag move: keep Y, apply the XZ offset captured at drag start.
Vec3 planeDragPosition(const Vec3& hitPoint, const Vec3& dragOffset,
                       float preservedY);

constexpr float kGizmoHandleRadius = 0.25f; // shared by pick test + visual
```
`planeDragPosition` = `{hit.x + offset.x, preservedY, hit.z + offset.z}`.
Drag-start offset (computed by the caller) = `{objPos.x - hit.x, 0,
objPos.z - hit.z}`. Both helpers are trivial and fully unit-testable.

### 2. Drag state — owned by `Application` (`core/application.{h,cpp}`)
Selection stays in `Editor`; the **drag state lives in `Application`** because all
the input events, camera, renderer (aspect/ray) math already live there and
`Editor` has no access to the camera. This avoids plumbing camera/renderer into
`Editor`. (Deliberate, documented deviation from the "editor-owned" suggestion —
the alternative needs more wiring for no benefit at this scope.)

Private members:
```cpp
bool   gizmoDragging_   = false;
Vec3   gizmoDragOffset_ = {};
float  gizmoDragPlaneY_ = 0.0f;
GameObject* gizmoDragTarget_ = nullptr; // mirrors editor_->selected()
```
Extract the existing NDC→ray block into `Ray rayFromPixel(float x, float y)`
so button-down and drag both reuse it.

`onMouseButton(0, true, x, y)`:
1. Build ray.
2. If `editor_->selected()` has a `MeshRenderer`, test the ray against the
   handle sphere at the object's world origin:
   `intersectRaySphere(rayOrigin - objPos, rayDir, kGizmoHandleRadius, t)`.
   On hit → begin drag: `gizmoDragPlaneY_ = objPos.y`, intersect the horizontal
   plane at that Y, capture `gizmoDragOffset_`, set `gizmoDragging_ = true`,
   `gizmoDragTarget_ = selected`. **Return without reselecting.**
3. Otherwise keep current behavior: `pickObject` + `setSelected`.

`onMouseButton(0, false, x, y)` (mouse up): clear `gizmoDragging_`.

New `onLeftMouseMove(float x, float y)`: if `gizmoDragging_`, build ray,
`intersectRayHorizontalPlane(.., gizmoDragPlaneY_, hit)`, then
`target->transform().position = planeDragPosition(hit, gizmoDragOffset_,
gizmoDragPlaneY_)`. Guard with the existing `ImGui::WantCaptureMouse` check.

### 3. Platform wiring — `platform/window.mm`
Add `mouseDragged:` → `onLeftMouseMove(pt.x, pt.y)` and
`mouseUp:` → `onMouseButton(0, false, pt.x, pt.y)`, mirroring the existing
`mouseDown:`/`rightMouseDragged:` conversion of `locationInWindow`.

### 4. Gizmo visual — `Scene::buildRenderCommands` (`scene/scene.cpp`)
After the selection highlight, if `selected` has a `MeshRenderer`, emit one
small **opaque yellow sphere** at the object's world origin:
`translate(worldPos) * scale(kGizmoHandleRadius)`, `ShapeType::Sphere`. Visual
radius == pick radius (shared `kGizmoHandleRadius`) so the handle you see is the
handle you click. Reuse the existing `Sphere` mesh; no new shape type.

## Tests (deterministic, no screenshots)
- **`src/tests/test_picking.cpp`** (`runTestPicking`, plain `assert`):
  - `intersectRayHorizontalPlane`: hit at arbitrary Y returns correct hitPoint;
    parallel ray (dir.y ≈ 0) misses; hit behind origin (t < 0) misses.
  - `planeDragPosition`: preserves Y; applies XZ offset; with the start offset it
    reproduces the original object position (round-trip), and a moved hitPoint
    shifts X/Z by the same delta.
  - Optional integration: gizmo-handle hit test — ray at object origin hits the
    `kGizmoHandleRadius` sphere; ray to empty space misses.
- **`src/tests/test_renderer.cpp`**: selecting a `MeshRenderer` object adds the
  gizmo sphere command (assert an extra opaque-yellow `Sphere` is present);
  update any exact render-command counts to account for the new command. (This
  file is already modified by the 0.10 highlight work — fold the count change in.)

## Manual smoke case (interactive, user-run)
Run `./build/debug/sandbox`, then:
1. Click an object → it selects (highlight + yellow handle appear).
2. Click the yellow handle and drag → the object follows the cursor across the
   XZ plane; its Y is unchanged.
3. Release → the object stays put.
4. Click empty space → selection clears, the handle disappears, and **no drag
   lingers** (mouse-up always clears `gizmoDragging_`/`gizmoDragTarget_`; a fresh
   click with nothing selected falls through to `pickObject` → `setSelected`).

## Out of scope (explicitly not doing)
Rotation, scale, axis handles/arrows, local/world modes, snapping,
multi-selection, undo/redo, serialization. No generalized gizmo framework — just
the four drag-state fields above.
