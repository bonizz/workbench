# Milestone 0.7 Plan — Primitive Mesh Library

Status: APPROVED. Implemented exactly as specified below.

Supersedes the recommendation in `docs/plans/0.7_next_milestone_recommendation.md`
(which is now the "why"; this is the "how").

## Goal

Give `MeshRenderer` **real, selectable geometry**. Today every object that has a
`MeshRenderer` renders as the same hardcoded cube; the `mesh` field is stored,
serialized, and round-tripped but **completely ignored by the renderer**. This
milestone wires that field to an actual primitive set (cube, sphere, plane),
generated procedurally on the CPU, drawn from static per-shape GPU buffers,
authored via a name-based command and the inspector, asserted deterministically,
surfaced in DebugState, and serialized backward-compatibly.

This turns a latent inconsistency (a data field the renderer can't honor) into a
feature, and lays the geometry groundwork that a future lighting milestone builds
on (normals get added to the same procedural vertices).

## Non-goals (explicit, per the approved constraints)

- **No asset pipeline.** Geometry is generated in code, like the cube is today.
- **No model/mesh file loading** (`.obj`, `.gltf`, etc.).
- **No external geometry files.**
- **No lighting / normals / materials.** Flat per-instance color only.
- **No registry / reflection / factory.** `MeshRenderer` stays the one rendering
  component; shape selection is a plain enum, not a type hierarchy.
- **No new component type.** Still `MeshRenderer`. Keeps the per-type
  serialization fork at two types (conventions §8 / AGENTS.md).
- **No texture coordinates, UVs, or vertex attributes beyond position.**
- **No GPU instancing / batching.** One draw call per object, as today.

## 1. Architecture review (what exists today)

Facts that constrain this design (all verified against current source):

- **`MeshRenderer` is `{ std::string mesh = "cube"; simd::float4 color; }`**
  (`scene/mesh_renderer.h`). `clone()` is a copy-construct
  (`scene/mesh_renderer.cpp`). The `mesh` string is written/read by the
  serializer but **never consulted by the renderer**.
- **The renderer draws exactly one shape.** `Scene::buildRenderCommands`
  (`scene/scene.cpp:121`) calls `ctx.drawCube(obj->transform().worldMatrix(),
  mesh->color)` for every object with a `MeshRenderer`.
- **`RenderContext` has `drawCube` + `drawGround`** (`renderer/render_context.*`),
  each pushing a `RenderCommand { ShapeType type; Mat4 transform; float4 color;
  + grid fields }`. `ShapeType` is `{ Cube, Ground }` (`renderer/render_command.h`).
- **Cube geometry is generated per-draw into a dynamic vertex buffer.**
  `MetalRenderer::draw` (`renderer/metal_renderer.mm:286-309`) writes 8 colored
  vertices per cube into a shared `dynamicVertexBuffer` (capacity **512**
  vertices), indexed by a static 36-entry `cubeIndexBuffer`. **Color is baked
  per-vertex** and read by the shader.
- **The shader reads per-vertex color.** `assets/shaders/basic.metal`:
  `fragment_main` returns the interpolated `in.color`. `Vertex` is
  `{ position[3]; color[4] }` (28 bytes, `render_types.h`).
- **Cull mode is Metal's default (`MTLCullModeNone`)** — there is no
  `setCullMode` call — so single-sided geometry is visible from both faces.
- **Cube extents are `[-1, 1]`** (half-extent 1). `scene.create_cube` and the
  editor give new objects `scale = (0.5, 0.5, 0.5)`, so a default cube renders as
  a unit-sized box. New primitives must use the **same half-extent-1 convention**
  so `scale` behaves identically across shapes.
- **Serialization** writes `"mesh": "cube"` and parses it strictly (unknown
  *fields* are a load error; unknown *values* are currently accepted verbatim).
  `assets/scenes/default.scene` contains `"mesh": "cube"`.
- **DebugState** prints `comp->typeName()` per component, with a
  `(angularVelocity: …)` suffix for `RotateComponent`
  (`debug/debug_state.cpp:121-133`). Unit tests substring-search
  `components: MeshRenderer` (`tests/main.cpp:363`).
- **Build wiring is asymmetric and matters here.** The `sandbox` target *globs*
  `src/engine/renderer/**.cpp` and `src/engine/scene/**.cpp`. The `tests` target
  lists files **explicitly** and **compiles `mesh_renderer.cpp` but NOT
  `metal_renderer.mm`** (`premake5.lua:124,131`). Therefore **any geometry code
  that must be unit-tested cannot live in `metal_renderer.mm`.**

This last point is the pivot of the whole design (see §3/D3).

## 2. Supported primitive set

Three shapes, all procedurally generated, all centered at the origin with
**half-extent 1** (matching the existing cube so `Transform::scale` is uniform
across shapes):

| Shape  | Geometry | Verts | Indices | Notes |
|--------|----------|-------|---------|-------|
| `cube`   | The existing 8-corner box, `[-1,1]³` | 8 | 36 | Unchanged geometry; moved to a static buffer. The default. |
| `sphere` | UV sphere, radius 1, **16 slices × 12 stacks** | 221 | 1152 | Recognizable, cheap; counts are named constants, easy to tune. |
| `plane`  | Unit quad in the XZ plane at `y=0`, corners `(±1, 0, ±1)` | 4 | 6 | Single quad; visible from both sides (cull = none). |

Rationale for the choices:

- **Cube** is mandatory (it is what every existing scene/test uses).
- **Sphere** is the shape that proves the procedural generator does real work
  (rings/stacks, pole handling) and is the most visually distinct from a cube —
  the headline of the milestone.
- **Plane** is trivial but genuinely useful (floors, walls, cards) and exercises
  a degenerate, flat mesh. It is *not* the ground grid: the ground stays its own
  special `ShapeType::Ground` with its own shader; `plane` is an ordinary
  colored, transformable `MeshRenderer` primitive.

Index buffers stay **`uint16`** for all shapes (max 1152 indices ≪ 65 535).

Sphere parameters live as named constants in the generator
(`kSphereSlices = 16`, `kSphereStacks = 12`); changing them is a one-line edit
and a re-derivation of the expected vertex/index counts in the unit test.

## 3. Major design decisions (summary)

| # | Decision | Choice |
|---|----------|--------|
| D1 | Shape representation on `MeshRenderer` | Replace `std::string mesh` with `enum class MeshShape { Cube, Sphere, Plane }` + free `meshShapeToString` / `meshShapeFromString` (in `mesh_renderer.{h,cpp}`). JSON keeps the `"mesh"` **string** field for backward compatibility. |
| D2 | Geometry generation location | A **new plain-C++ module** `renderer/mesh_geometry.{h,cpp}` returning `MeshData { vector<Vertex>; vector<uint16_t> }`. Compiled by both `sandbox` (glob) and `tests` (explicit), so it is **unit-testable without Metal**. |
| D3 | GPU buffers | **Static per-shape** vertex+index buffers built once (lazily) from `mesh_geometry`. The per-draw dynamic vertex buffer is **removed**. |
| D4 | Per-instance color | Move color from per-vertex to a **fragment uniform** (one small `basic.metal` edit + a fragment buffer index). Required because static geometry is shared across instances. |
| D5 | `ShapeType` (renderer) | Extend to `{ Cube, Sphere, Plane, Ground }`. Generalize `RenderContext::drawCube` → `drawShape(ShapeType, transform, color)`; keep `drawGround`. |
| D6 | Authoring command | `component.set_mesh <name> <shape>` (name-based, matching the 0.5/0.6 convention). No `scene.create_sphere/plane` convenience commands this milestone. |
| D7 | Assertion | `assert.mesh <name> <shape>` (name-based), for deterministic shape testing without screenshots. |
| D8 | DebugState | Additive `(mesh: <shape>)` suffix on the `MeshRenderer` component line. Substring-safe. |
| D9 | Serialization | Save writes the shape string; load parses string→enum and **errors on an unknown value** (tightens today's lax acceptance). No-`mesh`-field objects still default to `cube`. |

Each is justified below; rejected alternatives are in §11.

## 4. `MeshRenderer` data model (D1)

```cpp
// scene/mesh_renderer.h
namespace scene {

enum class MeshShape { Cube, Sphere, Plane };

// Lowercase canonical names: "cube", "sphere", "plane".
const char* meshShapeToString(MeshShape shape);
// Returns false on an unrecognized name (out is unchanged).
bool meshShapeFromString(const std::string& name, MeshShape& out);

class MeshRenderer : public Component
{
public:
    const char* typeName() const override { return "MeshRenderer"; }
    std::unique_ptr<Component> clone() const override;

    MeshShape shape = MeshShape::Cube;          // was: std::string mesh = "cube"
    simd::float4 color = {1.0f, 1.0f, 1.0f, 1.0f};
};

} // namespace scene
```

- **Enum, not string**, so invalid shapes are unrepresentable at runtime and the
  renderer's `switch` is exhaustive (compiler warns on a missing case). The
  string↔enum mapping is centralized in **one** place
  (`mesh_renderer.cpp`) and reused by the serializer, command, and editor.
- `clone()` stays a copy-construct — it now copies the enum, which is correct and
  free.
- The **JSON field name remains `"mesh"`** with a **string value**
  (`"cube"`/`"sphere"`/`"plane"`). Existing files (`default.scene`, every
  `.wbs`-produced scene) load unchanged; the default is `Cube`, so omitting the
  field is still a cube.
- `meshShapeToString` returns a `const char*` to a static literal (no allocation);
  `meshShapeFromString` returns `bool` so callers (serializer, command) produce
  precise errors.

Both functions live in `mesh_renderer.cpp`, which the `tests` target already
compiles — so the mapping is unit-testable directly.

## 5. `RenderCommand` / `RenderContext` changes (D5)

`render_command.h`:

```cpp
enum class ShapeType { Cube, Sphere, Plane, Ground };   // + Sphere, Plane
```

`RenderCommand` is otherwise unchanged (it already carries `type`, `transform`,
`color`, and the ground-only grid fields).

`render_context.h` / `.cpp` — generalize the cube entry point:

```cpp
// Replaces drawCube. One mesh primitive, selected by shape.
void RenderContext::drawShape(ShapeType shape, const Mat4& transform,
                              const simd::float4& color);
// drawGround stays exactly as-is (it carries grid params).
```

`drawShape` asserts `shape != ShapeType::Ground` in debug (Ground goes through
`drawGround`), then pushes a `RenderCommand{shape, transform, color}`.

`Scene::buildRenderCommands` maps the component's `MeshShape` → `ShapeType` and
calls `drawShape`:

```cpp
if (MeshRenderer* mesh = obj->getComponent<MeshRenderer>()) {
    ctx.drawShape(toRenderShape(mesh->shape), obj->transform().worldMatrix(),
                  mesh->color);
}
```

`toRenderShape(MeshShape)` is a tiny local mapping in `scene.cpp` (3 cases). The
two enums are intentionally **separate**: `MeshShape` is the scene-authoring set
(no Ground); `ShapeType` is the renderer's command set (includes Ground, which is
not a mesh). Collapsing them would force "Ground" into the authoring surface.

The two `drawCube` callers update mechanically: `scene.cpp:131` (above) and the
unit test at `tests/main.cpp:82-83` (→ `drawShape(ShapeType::Cube, …)`).

## 6. `MetalRenderer` geometry-buffer changes (D2, D3, D4)

### 6a. New geometry module (`renderer/mesh_geometry.{h,cpp}`)

```cpp
// renderer/mesh_geometry.h
#include "renderer/render_types.h"   // Vertex
#include <cstdint>
#include <vector>

struct MeshData {
    std::vector<Vertex> vertices;     // position filled; color set to white (unused by shader)
    std::vector<uint16_t> indices;
};

MeshData makeCube();
MeshData makeSphere();   // uses kSphereSlices / kSphereStacks
MeshData makePlane();
```

- Plain C++/`simd` only — **no Metal, no Objective-C** — so it compiles into the
  `tests` target and is unit-testable.
- `Vertex.color` is filled with white and ignored by the (new) shader. It is kept
  only so the existing `Vertex` layout and pipeline vertex stride are unchanged;
  a future lighting milestone replaces that dead field with a normal.
- Winding is consistent CCW (front-facing) even though culling is off, so it's
  correct if culling is ever enabled.

### 6b. Static per-shape buffers (replaces the dynamic path)

`MetalRenderer::Impl` gains, and **drops**, buffers:

```cpp
// add: one vertex+index buffer per mesh shape, built once
id<MTLBuffer> cubeVertexBuffer = nil;   id<MTLBuffer> cubeIndexBuffer = nil;   // index buf already exists
id<MTLBuffer> sphereVertexBuffer = nil; id<MTLBuffer> sphereIndexBuffer = nil;
id<MTLBuffer> planeVertexBuffer = nil;  id<MTLBuffer> planeIndexBuffer = nil;
NSUInteger cubeIndexCount = 0, sphereIndexCount = 0, planeIndexCount = 0;

// remove: dynamicVertexBuffer, dynamicVertexOffset, DYNAMIC_VERTEX_CAPACITY,
//         allocateDynamicVerts, the in-draw 8-vertex cube generation.
```

A small helper builds a buffer pair from `MeshData` (lazily on first `draw`, like
the ground buffers are today):

```cpp
static void uploadMesh(id<MTLDevice> device, const MeshData& data,
                       id<MTLBuffer>& vbuf, id<MTLBuffer>& ibuf, NSUInteger& indexCount);
```

`cubePipeline` is **renamed `meshPipeline`** (same shader, now used by all three
mesh shapes). The ground keeps `gridPipeline`.

### 6c. Draw loop

The per-command `switch` collapses the three mesh shapes to one path that selects
the buffer pair, then draws indexed with the color uniform:

```cpp
case ShapeType::Cube:
case ShapeType::Sphere:
case ShapeType::Plane: {
    auto [vbuf, ibuf, indexCount] = selectMeshBuffers(cmd.type);   // picks the pair
    [enc setRenderPipelineState:impl_->meshPipeline];
    [enc setVertexBuffer:vbuf offset:0 atIndex:VertexBufferIndex::Vertices];
    ShaderUniforms u = {cmd.transform, view, projection};
    [enc setVertexBytes:&u length:sizeof(u) atIndex:VertexBufferIndex::Uniforms];
    [enc setFragmentBytes:&cmd.color length:sizeof(cmd.color)
        atIndex:MeshFragmentBufferIndex::Color];                   // per-instance color
    [enc drawIndexedPrimitives:MTLPrimitiveTypeTriangle
        indexCount:indexCount indexType:MTLIndexTypeUInt16
        indexBuffer:ibuf indexBufferOffset:0];
    break;
}
case ShapeType::Ground: { /* unchanged */ }
```

### 6d. Color as a fragment uniform (D4) — the one shader edit

Static geometry is shared by every instance of a shape, so per-instance color
**cannot** be baked into the vertices anymore. Color moves to a fragment buffer,
mirroring how the grid already passes per-draw fragment bytes.

`render_types.h` gains:

```cpp
enum class MeshFragmentBufferIndex : uint32_t { Color = 0 };
```

`assets/shaders/basic.metal` — **only `fragment_main` changes**:

```metal
fragment float4 fragment_main(VertexOut in [[stage_in]],
                              constant float4& uColor [[buffer(0)]])
{
    return uColor;
}
```

`vertex_main` is left untouched (it still copies the now-ignored vertex color;
harmless). `ShaderUniforms` and its `static_assert(… == 192)` are **untouched** —
color rides its own fragment buffer, not the uniforms struct. This keeps the
grid shader/pipeline and the uniform layout completely unaffected.

Net effect on the renderer: the dynamic-vertex machinery and its 512-vertex
overflow risk are **deleted**, replaced by build-once static buffers — fewer
moving parts, no per-frame vertex regeneration, and no capacity coupling to scene
size.

## 7. Scene serialization behavior (D9)

Format is **unchanged on disk**: still `"mesh": "<string>"`. Backward compatible
in both directions.

**Save** (`writeObject`, `scene_serializer.cpp`):

```cpp
out << "{\"type\": \"MeshRenderer\", ";
out << "\"mesh\": \"" << scene::meshShapeToString(mesh->shape) << "\", ";
out << "\"color\": [ … ]}";
```

**Load** (`parseComponent`): the local `std::string mesh = "cube"` stays as the
parse target; at construction it is converted to the enum, erroring on an
unrecognized value:

```cpp
if (type == "MeshRenderer") {
    scene::MeshShape shape;
    if (!scene::meshShapeFromString(mesh, shape)) {
        error = "Unknown mesh shape: " + mesh;
        return false;
    }
    auto mr = std::make_unique<MeshRenderer>();
    mr->shape = shape;
    mr->color = color;
    out.push_back(std::move(mr));
}
```

- Objects with **no** `mesh` field still default to `cube` (the local default is
  `"cube"`), so older/minimal files load unchanged.
- An **unknown value** (e.g. `"teapot"`) is now a clean load error instead of
  silently being stored and ignored. This tightens the strict-parser contract
  (conventions §6) consistently — unknown field *and* unknown mesh value both
  fail the load.
- `default.scene` (`"mesh": "cube"`) and every existing `.wbs`-authored scene
  round-trip byte-for-byte (default shape → `"cube"`).

## 8. Agent commands and assertions (D6, D7)

Two new **name-based** commands (consistent with rotator/assertion conventions;
stable across load). Both added to `commandTable()` so `agent.commands` /
`agent.help` discover them automatically (conventions §9).

| Command | Behavior |
|---------|----------|
| `component.set_mesh <name> <shape>` | Find object by name (error if missing). Error if it has no `MeshRenderer`. Parse `<shape>` via `meshShapeFromString` (error `Unknown mesh shape: <shape>` listing valid values). Set `mesh->shape`. Success: `Set <name> mesh to <shape>`. |
| `assert.mesh <name> <shape>` | Find object by name (assertion failure if missing). Assertion failure if no `MeshRenderer`, message `Object '<name>' has no MeshRenderer component.` (matching `assert.color`'s wording). Parse `<shape>` (usage error if invalid). Compare to `mesh->shape`; on mismatch, `makeAssertionFailure` with expected/actual shape names. |

Notes:

- `component.set_mesh` errors (missing object, no MeshRenderer, bad shape) are
  **command/usage errors**, not assertion failures (they don't set
  `lastAssertionFailure`) — same as `component.set_rotator`.
- `assert.mesh` "object missing" / "no MeshRenderer" are **assertion failures**
  (stop the script, set `lastAssertionFailure`) — same shape as `assert.color`.
- A bad `<shape>` token in `assert.mesh` is a usage error (the assertion can't be
  evaluated), consistent with how the numeric asserts treat unparseable values.
- **Deferred:** `scene.create_sphere` / `scene.create_plane` convenience
  creators. `scene.create_cube` + `component.set_mesh` is sufficient and more
  explicit; adding shape-specific creators is scope creep (§11).

Authoring a non-cube object from a script is therefore:

```text
scene.create_cube Ball
component.set_mesh Ball sphere
assert.mesh Ball sphere
```

## 9. Editor inspector changes

In `Editor::drawInspector`, the existing `MeshRenderer` block (`editor.mm:261-267`)
gains a shape selector above the color picker:

```cpp
if (MeshRenderer* mesh = selected_->getComponent<MeshRenderer>()) {
    ImGui::Text("MeshRenderer");

    const char* shapeNames[] = {"Cube", "Sphere", "Plane"};
    int current = static_cast<int>(mesh->shape);
    if (ImGui::Combo("Shape", &current, shapeNames, IM_ARRAYSIZE(shapeNames))) {
        mesh->shape = static_cast<scene::MeshShape>(current);
    }

    float color[4] = { … };           // unchanged color picker
    if (ImGui::ColorEdit4("Color", color)) { … }
}
```

- The `shapeNames[]` order must match the `MeshShape` enum order (Cube=0,
  Sphere=1, Plane=2). A `static_assert` on the count or a comment guards this;
  the canonical names still come from `meshShapeToString` everywhere
  non-interactive — the editor array is the one display-only duplication, kept
  local and obvious.
- The "Add MeshRenderer" buttons (`editor.mm:277`, and `create_cube` paths) are
  unchanged: new `MeshRenderer`s default to `Cube`.
- Live result: changing the combo re-renders the selected object as the new shape
  next frame (buildRenderCommands reads `shape`). No new editor state.

## 10. DebugState changes (D8)

One additive change in the component-listing loop (`debug_state.cpp:126-133`),
parallel to the existing `RotateComponent` suffix:

```cpp
out << comp->typeName();
if (auto* mr = dynamic_cast<const MeshRenderer*>(comp.get())) {
    out << " (mesh: " << scene::meshShapeToString(mr->shape) << ")";
} else if (auto* rot = dynamic_cast<const scene::RotateComponent*>(comp.get())) {
    out << " (angularVelocity: " << formatVec3(rot->angularVelocityEuler) << ")";

```

Substring stability (conventions §7): the line becomes
`components: MeshRenderer (mesh: cube)`. Existing tests search for
`components: MeshRenderer` (`tests/main.cpp:363`) and bare `MeshRenderer`
(`:330`) — both remain substrings, so they still pass. The exact round-trip test
compares `build()` output against itself, so it stays consistent. A **new** test
searches for `mesh: sphere`. No reorder/rename of existing output.

## 11. Tests, .wbs tests, and manual smoke tests

### Unit tests (`src/tests/main.cpp`)

Geometry generation is now testable without Metal (the whole point of D2):

- **`mesh_geometry` shape data**: `makeCube()` → 8 verts / 36 indices;
  `makeSphere()` → 221 verts / 1152 indices (derive from the named constants —
  the test re-computes `(stacks+1)*(slices+1)` and `stacks*slices*6` so tuning
  the constants updates the expectation in one place);
  `makePlane()` → 4 verts / 6 indices.
- **Geometry bounds**: every generated vertex of every shape lies within
  `[-1, 1]` on each axis (half-extent-1 contract). Sphere vertices satisfy
  `|length(pos) - 1| < epsilon`.
- **Index validity**: every index `< vertices.size()` for each shape.
- **`meshShapeToString` / `meshShapeFromString`** round-trip for all three names;
  `meshShapeFromString("teapot", …)` returns false and leaves `out` untouched.
- **`MeshRenderer` default** `shape == MeshShape::Cube`; `clone()` preserves a
  non-default shape (set to `Sphere`, clone, assert).
- **Serialization round-trip**: save a scene whose object is a `Sphere`
  MeshRenderer, load into a fresh scene, assert `shape == Sphere` and color
  preserved. Plus a **backward-compat** test: a hand-written JSON with
  `"mesh": "cube"` loads to `Cube`, and one **with no `mesh` field** loads to
  `Cube`. An **unknown value** (`"mesh": "teapot"`) fails `load` with
  `Unknown mesh shape: teapot`.
- **`component.set_mesh`** command: success; error on missing object; error on
  object without MeshRenderer; error on unknown shape.
- **`assert.mesh`** command: success; failure on mismatch (sets
  `lastAssertionFailure`); assertion failure on missing object and on
  no-MeshRenderer (message matches `assert.color`'s wording).
- **Command discovery**: `component.set_mesh` and `assert.mesh` appear in
  `agent.commands` and have `agent.help` entries (extend the existing
  "new commands appear in discovery" test).
- **DebugState**: dump contains `mesh: sphere` for a sphere object and still
  contains `components: MeshRenderer`.
- **`drawShape` mapping**: build a scene with cube/sphere/plane objects, call
  `buildRenderCommands`, assert the emitted `RenderCommand.type`s match (this
  covers `toRenderShape` and the `MeshShape`→`ShapeType` mapping without Metal,
  since `RenderContext` is in the tests target).

### `.wbs` suite tests (`assets/tests/`)

- **`mesh_shapes.wbs`** — authoring + assertion across all three shapes:

### Manual smoke tests (the renderer/shader path, which the suite cannot cover)

The Metal draw path and the `basic.metal` color-uniform edit are **not** exercised
by the unit suite (no `.mm` in `tests`, screenshots are async). Verify visually:

1. `./build/debug/sandbox` — create three objects, set them to cube / sphere /
   plane via the inspector Shape combo; confirm each renders as the right shape
   and that **per-object color still works** (proves the fragment-color uniform).
2. Set distinct colors on each; confirm colors are independent (proves color is

3. `render.capture mesh_smoke.png` (or a repro bundle) and eyeball the PNG.
4. Load `default.scene`; confirm the existing cube still renders identically
   (backward-compat of the cube path + serialization).

These steps are listed in the milestone's acceptance criteria (§13) as the manual
gate before merge.

## 12. Risks and alternatives considered

### Risks / edge cases

1. **Shader/renderer path is untestable in the suite.** Highest risk. The color
   move to a fragment uniform and the multi-shape draw loop can only be validated
   by the manual smoke tests (§11). Mitigation: keep the shader edit minimal
   (one `fragment_main` signature), keep all *logic* (geometry, mapping,
   serialization) in testable plain-C++ modules, and gate merge on the §13 manual
   checks.
2. **Color regression.** Moving color from per-vertex to a uniform is the one
   behavioral change to existing cube rendering. If the fragment buffer index or
   `setFragmentBytes` size is wrong, cubes render black/garbage. Covered by smoke
   test #1–#2 (per-object color must still work).
3. **Enum/array order drift in the editor combo.** `shapeNames[]` must match
   `MeshShape` order. Guarded by a comment + the values flowing through
   `static_cast`; the canonical mapping is still `meshShapeToString` everywhere
   else. Low impact (display only).
4. **Sphere vertex count vs. old dynamic capacity.** Irrelevant now — static
   buffers are sized per shape; the 512-vertex limit is deleted. A scene with
   many spheres just reuses one 221-vertex buffer.
5. **Backward compatibility.** Default shape is `Cube`; the `"mesh"` field stays a
   string; missing field ⇒ cube. `default.scene` and all existing `.wbs` scenes
   load unchanged. Covered by serialization unit tests.
6. **Build wiring.** `mesh_geometry.cpp` must be added to the **`tests` target's
   explicit file list** (the `sandbox` glob picks it up automatically). Forgetting
   this fails the geometry unit tests to compile — caught immediately by
   `./test.sh`.
7. **`drawCube` rename ripple.** Two callers (`scene.cpp`, one unit test) — both
   updated mechanically. No other consumers.
8. **Plane visibility from below.** Cull mode is `none` today; a single-sided
   quad is visible from both faces. If culling is ever enabled, the plane needs
   back faces — noted in the geometry source.

### Alternatives considered and rejected

- **Keep generating geometry per-draw in a dynamic buffer (extend today's cube
  pattern).** No shader change, but: (a) regenerates sphere vertices 60×/s,
  (b) couples a fixed buffer capacity to scene size (a few spheres overflow 512),
  (c) doesn't generalize to the lighting milestone (which wants static geometry +
  uniforms). Rejected in favor of static buffers; the one-line shader edit is the
  cheaper long-term cost.
- **Put geometry generation in `metal_renderer.mm`.** Simpler include graph, but
  the `tests` target doesn't compile the `.mm`, so the generator would be
  **untestable**. Rejected — testability is an explicit milestone goal. Hence the
  separate `mesh_geometry` TU (D2).
- **Keep `mesh` as a `std::string` and validate strings everywhere.** Smaller
  diff, but stringly-typed: invalid shapes representable at runtime, non-exhaustive
  renderer `switch`, validation logic duplicated at every set site. Rejected in
  favor of an enum with a single mapping point.
- **Add `color` to the shared `ShaderUniforms` struct.** Would change the struct,
  its `static_assert`, *and* require keeping `grid.metal`'s `Uniforms` in lockstep.
  Rejected in favor of a dedicated fragment buffer (smaller blast radius; mirrors
  the grid's existing per-draw fragment-bytes pattern; leaves the uniform layout
  and grid pipeline untouched).
- **Merge `MeshShape` into the renderer's `ShapeType`.** Would drag `Ground`
  (not a mesh) into the authoring/serialization surface. Rejected; the two enums
  describe different domains (scene authoring vs. render command).
- **`scene.create_sphere` / `scene.create_plane` convenience creators.** Handy
  but redundant with `create_cube` + `set_mesh`, and they multiply the creation
  surface. Deferred (§14).
- **Normals now, to "save a step" before lighting.** Violates the no-lighting
  constraint and adds an unused vertex attribute. Rejected; the lighting milestone
  extends `Vertex`/`mesh_geometry` deliberately.

## 14. Out of scope (explicitly deferred)

- `scene.create_sphere` / `scene.create_plane` convenience creators.
- Any additional primitives (cylinder, cone, capsule, torus).
- Per-vertex normals, lighting, materials, textures/UVs.
- Mesh/model file loading or an asset pipeline.
- GPU instancing / draw-call batching.
- Configurable sphere tessellation from the editor or commands (constants only).
- Wireframe / debug-draw rendering modes.
- A component/serializer registry (still the explicit two-type fork).

## 15. Acceptance criteria

The milestone is complete when:

1. `./test.sh` passes, including all new `mesh_geometry`, serialization,
   `component.set_mesh`, `assert.mesh`, and DebugState unit tests.
2. `./build/debug/sandbox --run-tests --exit` reports the new `.wbs` suite tests
   (`mesh_shapes`, `mesh_persistence`) as PASS and exits `0`.
3. `assert.mesh` and `component.set_mesh` appear in `agent.commands` and have
   `agent.help` entries.
4. `default.scene` and existing `.wbs`-authored scenes load and render unchanged
   (cube path is byte-for-byte compatible; unknown mesh values now error).
5. **Manual smoke (§11):** in the interactive editor, an object can be switched
   between cube / sphere / plane via the Shape combo and renders as that shape,
   with **independent per-object color** preserved (validates the fragment-color
   uniform and the static-buffer draw path).
6. A `render.capture` / repro bundle of a mixed cube+sphere+plane scene shows the
   three distinct shapes.
7. The dynamic vertex buffer and its overflow path are removed from
   `metal_renderer.mm` (no `DYNAMIC_VERTEX_CAPACITY` remains).
