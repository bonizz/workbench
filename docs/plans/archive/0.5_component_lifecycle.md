# Milestone 0.5 Plan — Component Lifecycle + RotateComponent

Status: IMPLEMENTED. (Status line corrected on archival — shipped as
`rotate_component.{h,cpp}` and the `onStart`/`onUpdate` lifecycle.)

## Goal

Turn `Component` from inert data into runtime behavior via a minimal, deterministic
lifecycle (`onStart` / `onUpdate`), and prove it with a second concrete component,
`RotateComponent`. No ECS, no registry, no reflection, no scripting language.

## Constraints honored

- No component registry / factory / reflection.
- No refactor of the command system (new commands are added alongside existing ones).
- Camera stays a `GameObject` subclass; not migrated.
- `MeshRenderer` behavior unchanged.
- Test suite stays green.
- Focus maintained: one lifecycle, one new component, the commands/serialization/debug/editor/test/doc plumbing for them.

---

## 1. Component lifecycle design

### Hooks on `scene::Component`

Add two virtual methods with default no-op implementations:

```cpp
virtual void onStart() {}
virtual void onUpdate(float /*dt*/) {}
```

`typeName()` and `clone()` remain as today. No other changes to the Component base.

### `Scene::update(float dt)`

New public method on `Scene`:

```cpp
void Scene::update(float dt);
```

Behavior, in exact order:

1. Iterate `objects_` in **vector order** (which is creation order, which is `ObjectId` order, since ids are allocated sequentially and objects are `push_back`ed).
2. Skip objects where `!obj->active()` (consistent with `Scene::buildRenderCommands`).
3. For each remaining object, iterate its `components_` in **add order** (vector order).
4. For each component:
   - If it has not been started yet, call `onStart()` and mark it started.
   - Then call `onUpdate(dt)`.

So `onStart` is called **lazily, on the component's first `Scene::update`**, immediately
before its first `onUpdate`. This is what makes scene load / test setup safe: building a
scene (create, add components, set values) never triggers behavior. Behavior only runs
when the simulation is explicitly advanced.

### Documented update order (to be written into `docs/components.md`)

- Objects: creation order == `ObjectId` ascending == `objects_` vector order.
- Components per object: add order == `components_` vector order.
- Per component: `onStart()` once (before its first `onUpdate`), then `onUpdate(dt)` every subsequent update.
- Inactive objects are skipped entirely (no `onStart`, no `onUpdate`).
- `Camera` has no components, so it is unaffected by `Scene::update`. `Camera::update` continues to be called separately by `Application` (unchanged).

## 2. Where onStart state is tracked

On the `Component` itself, as private state:

```cpp
private:
    bool started_ = false;
    friend class ::Scene;   // Scene is the sole driver of the lifecycle
```

Rationale:

- It is per-component state, so it belongs on the component.
- `Scene` is the only thing that drives the lifecycle; friending `Scene` keeps the
  transition explicit and avoids a public `markStarted()` that anything could call.
- `clone()` constructs a brand-new component, so `started_` defaults to `false` on
  duplicates — a duplicated spinning object starts its own lifecycle on the next update.
  This is the desired, least-surprising behavior.
- Loaded components (deserialized) are fresh instances, so `started_ == false`; they
  start on the first update after load.

No separate "started set" or per-Scene bookkeeping. One bool per component.

## 3. How `Scene::update` integrates with Application and `sim.step`

### The determinism problem

The test suite and `--run-script` automation must be driven **only** by `sim.step` with
fixed `dt`. If the interactive app also auto-advances the simulation every frame during
automation, a spurious `Scene::update(realFrameDt)` would non-deterministically rotate
objects between `sim.step` calls and break tests.

### Resolution: two modes, decided once at startup

Add a member to `Application`:

```cpp
bool liveSimulation_ = false;   // true only in interactive (no automation) mode
```

Set it once in `Application::init()` (after `cliOptions_` is known — `main.mm` calls
`setCliOptions` before `init`):

```cpp
liveSimulation_ = cliOptions_.runScript.empty()
               && cliOptions_.bundleName.empty()
               && !cliOptions_.runTests;
```

In `Application::onUpdate`, the simulation advance becomes:

```cpp
if (liveSimulation_) {
    scene_->update(time_.deltaTime());
}
scene_->camera().update(time_.deltaTime(), input);   // unchanged
```

Effect:

- **Interactive** (`./build/debug/sandbox`): `liveSimulation_ == true`; components animate every frame; `RotateComponent` visibly spins in the editor. `Camera::update` unchanged.
- **Automation** (`--run-script`, `--run-tests`, `--bundle`): `liveSimulation_ == false`; the simulation advances **only** when a script calls `sim.step`. Deterministic.
- `runTestSuite()` runs inside `onUpdate`'s `Pending` state and recreates scenes per test; with `liveSimulation_ == false` there is no frame-driven update polluting the tests. `sim.step` in the `.wbs` script is the sole driver.

`Camera::update` still runs every frame in all modes (harmless in automation: no input, no movement). Left unchanged to avoid touching camera behavior.

### `sim.step` command

```
sim.step [dt]
```

- Calls `ctx.scene.update(dt)`.
- `dt` is optional; default `1.0f / 60.0f` (documented). Tests always pass an explicit value (e.g. `1.0`) for determinism.
- Output: `Stepped simulation by <dt>s` (so transcripts are readable).
- Available everywhere commands are (Agent Console, scripts, CLI). In interactive mode it would double-advance on top of the live update; documented as "primarily for automation/tests."

`sim.step` is the **only** new command that touches the update loop. No `sim.start`/`sim.pause` this milestone.

## 4. RotateComponent implementation

New files: `src/engine/scene/rotate_component.h` / `.cpp`. Mirrors `mesh_renderer`.

```cpp
// rotate_component.h
#pragma once
#include "scene/component.h"
#include "core/math.h"
#include <memory>

namespace scene {

class RotateComponent : public Component
{
public:
    const char* typeName() const override { return "RotateComponent"; }
    std::unique_ptr<Component> clone() const override;
    void onUpdate(float dt) override;

    // Degrees per second, per Euler axis.
    Vec3 angularVelocityEuler{0.0f, 0.0f, 0.0f};
};

} // namespace scene
```

```cpp
// rotate_component.cpp
std::unique_ptr<Component> RotateComponent::clone() const
{
    auto c = std::make_unique<RotateComponent>(*this);
    // started_ is false on the fresh copy (base defaults). Intentional.
    return c;
}

void RotateComponent::onUpdate(float dt)
{
    GameObject* go = gameObject();
    if (!go) return;
    Vec3& rot = go->transform().rotation;          // radians (existing convention)
    rot.x += angularVelocityEuler.x * kDegToRad * dt;
    rot.y += angularVelocityEuler.y * kDegToRad * dt;
    rot.z += angularVelocityEuler.z * kDegToRad * dt;
}
```

### Units (important, documented)

- `Transform::rotation` is stored in **radians** (existing convention per `docs/scenes.md` and `DebugState`'s rad→deg display).
- `RotateComponent::angularVelocityEuler` is in **degrees per second** per the milestone spec.
- `onUpdate` converts deg/s → rad/s, scales by `dt`, and adds to the radian rotation.
- `assert.rotation` reports **degrees** (converts rad→deg for comparison), matching `DebugState`'s existing display.

### Deg/rad helper

`core/math.h` currently has no deg/rad constants (`command.cpp` and `debug_state.cpp`
each define their own `kRadToDeg`). I will add one shared constant to `math.h`:

```cpp
constexpr float kPi = 3.14159265f;
constexpr float kDegToRad = kPi / 180.0f;
constexpr float kRadToDeg = 180.0f / kPi;
```

and leave the existing local constants in place (not refactoring those files this milestone; just providing the shared one for the new component). Low risk.

### New agent commands (name-based, per spec)

| Command | Behavior |
|---|---|
| `component.add_rotator <name>` | Find object by name; error if not found; error if it already has a `RotateComponent`; add a default `RotateComponent` (velocity 0,0,0). |
| `component.set_rotator <name> <x> <y> <z>` | Find object by name; error if not found or has no `RotateComponent`; set `angularVelocityEuler`. Values in deg/s. |
| `sim.step [dt]` | `ctx.scene.update(dt)`; default dt `1/60`. |
| `assert.rotation <name> <x> <y> <z> [tolerance]` | Find object by name; convert its `Transform::rotation` rad→deg; compare each axis to `x y z` within `tolerance` (default `0.01` deg). On mismatch, assertion failure (sets `lastAssertionFailure`, stops script). |

Rationale for name-based new commands: tests must not depend on runtime `ObjectId`s (ids are reassigned on load and differ across runs). The existing `component.add_mesh_renderer <id>` stays id-based and unchanged (do not refactor the command system). This produces a small, intentional inconsistency (new rotator commands use names, old mesh command uses id) which is explicitly within scope and will be noted in docs.

`assert.rotation` failure uses the existing `makeAssertionFailure` path, so it integrates with the existing assertion-failure/`lastAssertionFailure`/script-stop behavior — no new failure machinery.

### `agent.help` / `agent.commands`

The four new commands are added to the static `commandTable()` in `command.cpp` with `usage`/`description`/`example`, so discovery is automatic. No structural change to the command system.

## 5. Serialization approach

Keep the existing **explicit per-type branch** design (no registry). `SceneSerializer` currently branches on `MeshRenderer` via `dynamic_cast` in save and on `type == "MeshRenderer"` in load. Add a parallel `RotateComponent` branch.

### Save (`scene_serializer.cpp`)

In the per-component output loop, alongside the existing `MeshRenderer` cast:

```cpp
} else if (auto* rot = dynamic_cast<const RotateComponent*>(comp.get())) {
    if (!firstComp) out << ", ";
    firstComp = false;
    out << "{\"type\": \"RotateComponent\", ";
    out << "\"angularVelocity\": ["
        << rot->angularVelocityEuler.x << ", "
        << rot->angularVelocityEuler.y << ", "
        << rot->angularVelocityEuler.z << "]}";
}
```

### Load (`Parser::parseComponent`)

Add an `else if (type == "RotateComponent")` branch that parses the `angularVelocity` array (reusing the existing `parseFloat3` helper into a `Vec3`) and constructs the component. Unknown field handling: keep the existing strict "unknown field = error" behavior; `angularVelocity` is the only new field.

### Format

```json
{
  "type": "RotateComponent",
  "angularVelocity": [0.0, 90.0, 0.0]
}
```

Matches the format in the milestone spec exactly. `clone()` preserves it on duplicate.

This is the second serialized component type, so it concretely surfaces the "per-system type fork" discussed in the architecture review (serializer, editor, commands, debug each branch on type). That is **acceptable and intentional** for this milestone — the spec forbids a registry, and the fork stays shallow and explicit until a third type justifies a refactor.

## 6. DebugState support

In `DebugState::build`, the component-listing loop currently prints only `typeName()`.
Add per-component detail for `RotateComponent`:

```cpp
for (const auto& comp : obj->components()) {
    if (!first) out << ", ";
    first = false;
    out << comp->typeName();
    if (auto* rot = dynamic_cast<const RotateComponent*>(comp.get())) {
        out << " (angularVelocity: "
            << formatVec3(rot->angularVelocityEuler) << ")";   // degrees, already deg
    }
}
```

`angularVelocityEuler` is already in degrees, so no conversion needed (unlike rotation).
Format stays stable for `MeshRenderer` (existing tests check `components: MeshRenderer`).

### Unit tests (`src/tests/main.cpp`)

Add focused unit assertions (mirroring existing style) for:

- `RotateComponent` default `angularVelocityEuler == {0,0,0}` and `typeName() == "RotateComponent"`.
- `clone()` preserves `angularVelocityEuler` and produces a not-started copy.
- `Scene::update(dt)` advances rotation: create object, add `RotateComponent`, set velocity `{0,90,0}`, `scene.update(1.0f)`, assert `rotation.y` ≈ `kDegToRad * 90` within epsilon.

- `sim.step` command: `executeCommand("sim.step 1.0", ctx)` advances a rotator's object.
- `component.add_rotator` / `component.set_rotator` commands: success + error cases (missing object, already-has, no-rotator-on-set).
- `assert.rotation` command: success and failure (failure sets `lastAssertionFailure`).
- Serialization: save a scene with a `RotateComponent`, load into a fresh scene, verify `angularVelocityEuler` round-trips.

These keep the existing `tests` executable as the source of truth and do not rely on the async screenshot path.

## 10. Risks and edge cases

1. **Determinism pollution in tests.** Highest risk. Mitigation: `liveSimulation_` is false whenever any automation flag is set, and `sim.step` is the only updater in automation. Covered by the `rotator.wbs` test, which would flake immediately if a spurious update occurred.
2. **Float precision in `assert.rotation`.** deg→rad→deg round-trip of `90.0` is within ~1e-5. Default tolerance `0.01` deg is safe; tests pass explicit values and don't rely on the default for boundary cases. Optional tolerance arg lets future tests tighten/loosen.
3. **`onStart` on cloned/loaded components.** `clone()` and deserialization produce fresh instances with `started_ == false`; they start on the next update. Desired behavior; covered by a unit test for `clone()`.
4. **Inactive objects.** Skipped entirely in `Scene::update` (no `onStart`, no `onUpdate`), matching `buildRenderCommands`. If activated later, `onStart` fires on the first update after activation. Documented.
5. **Component added mid-simulation.** Gets `onStart` on the next `Scene::update`, then `onUpdate`. Natural and deterministic.
6. **`sim.step` default `dt`.** Chosen as `1/60` s. Tests always pass explicit `dt`, so the default only affects manual console use. Documented.
7. **Editor live update vs. `sim.step` double-advance.** In interactive mode both can run. Documented as expected: `sim.step` is primarily for automation; in the editor the sim already advances per frame. No correctness issue, only a UX nuance.
8. **Second serialized type surfaces the type-fork.** Intentional and in-scope (no registry). Keeps the fork shallow and explicit; a future milestone can consolidate when a third type appears.
9. **`kDegToRad` constant addition.** Trivial, additive; existing local `kRadToDeg` constants are left untouched to keep the diff focused.
10. **Camera untouched.** `Camera` has no components, so `Scene::update` never touches it; `Camera::update` stays called by `Application`. No migration, per constraints.
