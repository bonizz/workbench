# Milestone 0.6 Plan — Transform Hierarchy

Status: IMPLEMENTED. (Status line corrected on archival — shipped in commit
`c422a43`, "feat(scene): transform hierarchy (Milestone 0.6)".)

## Goal

Give GameObjects parent/child relationships, derive **world transforms** from
**local transforms** through the hierarchy, render with world transforms,
serialize the hierarchy, expose it to the agent command surface, show it clearly
in DebugState and the editor, and cover it with deterministic tests.

No ECS, no reflection, no dynamic registration. The existing GameObject +
Component architecture, the Scene ownership model, the explicit per-type
serializer, and the name-based command conventions are all preserved.

## Constraints honored

- No ECS. Hierarchy is an explicit GameObject/Scene feature.
- No reflection / registry / factory. The serializer stays explicit.
- GameObject + Component architecture preserved. `Transform` stays the
  per-object transform; `Component` is unchanged.
- Scene stays the single owner of object lifetime (`objects_`).
- Existing tests stay green. Existing scenes/files load unchanged.
- Deterministic behavior for scripts, assertions, repro bundles, and the suite.
- Name-based addressing for all new commands (stable across load, matching the
  rotator/assertion conventions from 0.5).

---

## 1. Architecture review (what exists today)

Relevant facts that constrain this design:

- **Ownership & iteration.** `Scene` owns `std::vector<std::unique_ptr<GameObject>> objects_`
  plus a `Camera*` (the camera is a `GameObject` subclass pushed into `objects_`).
  `Scene::update` and `Scene::buildRenderCommands` iterate `objects_` in vector
  (== creation == `ObjectId` ascending) order. This order is the documented
  determinism contract and **must not change**.
- **Transform is local-only.** `Transform` is a plain struct of
  `position / rotation (radians) / scale`, plus `localMatrix()` =
  `T * Ry * Rx * Rz * S`. There is no world matrix, no parent, no children.
- **Rendering uses localMatrix directly.** `Scene::buildRenderCommands` calls
  `ctx.drawCube(obj->transform().localMatrix(), mesh->color)`. This is the only
  render consumer of `localMatrix` (the only other caller is a unit test).
- **Components mutate local fields.** `RotateComponent::onUpdate` adds to
  `go->transform().rotation`. `Scene::update(dt)` drives `onStart`/`onUpdate`
  in vector order; inactive objects are skipped.
- **Simulation determinism.** Interactive mode auto-advances per frame;
  automation mode (`--run-script`/`--run-tests`/`--bundle`) advances **only**
  via `sim.step`, so the simulation is deterministic. `Scene::update(dt)` is the
  single driver.
- **Serialization is explicit & name-free for relationships.** Objects serialize
  `name/position/rotation/scale/components`. `ObjectId`s are NOT serialized
  (reassigned on load). The tokenizer rejects unknown fields.
- **DebugState is a stable, substring-searchable text dump**, sorted by id, with
  per-object blocks (`[id] name`, `position:`, `rotation:` (deg), `scale:`,
  `components:`). Several unit tests assert substrings of it.
- **Commands are mixed id-based (legacy) and name-based (new).** The newer
  commands (rotator, all `assert.*`) are name-based because ids are not stable
  across load.
- **Camera is special.** It is a root object, marked inactive, updated via its
  own `Camera::update`, never serialized, and cannot be deleted/duplicated.
- **Build.** `premake5.lua` globs `src/engine/scene/**.cpp` for the `sandbox`
  target and lists files explicitly for `tests`. **No new translation units are
  introduced by this plan** (all edits are to existing `.cpp`/`.h`), so the
  build files need no changes and `./build.sh` / `./test.sh` regenerate cleanly.

## 2. Major design decisions (summary)

| # | Decision | Choice |
|---|---|---|
| D1 | Hierarchy storage | Flat `objects_` + non-owning `parent_` / `children_` on each GameObject. Scene is the sole mutator. |
| D2 | World matrix | Derived **cache** on `Transform`, recomputed by an explicit `Scene::updateWorldTransforms()` (DFS from roots). No dirty flags. |
| D3 | Authoring semantics | Author fields (`position/rotation/scale`) are **LOCAL**. All existing transform commands/inspector keep editing local. World is derived. |
| D4 | Parenting API | `Scene::setParent(child, parent)` — Scene validates cycles, camera exclusion, and children-vector invariants. |
| D5 | Deletion | **Cascade** delete the whole subtree. |
| D6 | Duplication | **Cascade** duplicate the whole subtree (preserve local transforms + components + child names; root gets ` Copy`). |
| D7 | Serialization | **Nested `children` array** in the object JSON (tree). Backward compatible: scenes with no children serialize identically. |
| D8 | DebugState | Keep per-object detail blocks (substring-safe); add a `Hierarchy:` tree section; add `parent:` and `worldPosition:` lines per object. |
| D9 | Freshness contract | `updateWorldTransforms()` called by `buildRenderCommands`, end of `update(dt)`, `setParent`, `DebugState::build`, and the world-reading commands. |
| D10 | Camera in hierarchy | Excluded: cannot be a parent or a child. Keeps its separate update path simple. |

Each is justified in the sections below, with rejected alternatives in §11.

## 3. Parenting model

### Storage

`GameObject` gains non-owning hierarchy pointers. `Scene` remains the sole owner
of object lifetime and the sole mutator of these pointers.

```cpp
class GameObject {
    // ... existing members ...
public:
    GameObject* parent() const { return parent_; }
    const std::vector<GameObject*>& children() const { return children_; }
private:
    friend class Scene;
    GameObject* parent_ = nullptr;
    std::vector<GameObject*> children_;  // insertion order; deterministic
};
```

Rationale:

- Flat storage preserves `objects_` iteration order (the determinism contract).
  A tree-ownership model (GameObject owning children via `unique_ptr`) would
  reorder iteration, complicate `findObjectById`/delete/duplicate, and break the
  single-owner invariant. Rejected (§11).
- Non-owning pointers mirror the existing `Camera*` pattern and keep lifetime
  trivial: deleting an object removes it (and its subtree) from `objects_`; there
  are never dangling pointers because all mutations go through Scene.
- `friend class Scene` (already used in 0.5 for `Component::started_`) keeps the
  pointers private with Scene as the explicit sole mutator.

### API

```cpp
class Scene {
public:
    // Returns false (and leaves the hierarchy unchanged) on:
    //   - null child, or child is the camera
    //   - parent is the camera
    //   - child == parent
    //   - parent is a descendant of child (would create a cycle)
    // Passing parent == nullptr detaches child (makes it a root).
    // Idempotent: setting the current parent again is a success no-op.
    // Maintains children_ invariants on old and new parent, then refreshes
    // world transforms.
    bool setParent(GameObject* child, GameObject* parent);
};
```

Invariants maintained by `setParent` (and relied on everywhere else):

1. `A->parent_ == B`  ⇔  `B->children_` contains `A` exactly once.
2. No cycles (the graph is a forest of trees rooted at `parent_ == nullptr`).
3. The camera has `parent_ == nullptr` and `children_` empty.
4. `children_` order is the order children were attached (insertion order).

Cycle guard: `isDescendant(possibleChild, possibleAncestor)` walks
`possibleChild->children_` recursively; if `possibleAncestor` is found, reject.
`setParent(child, parent)` calls `isDescendant(parent, child)` — if `child` is an
ancestor of `parent`, parenting would create a cycle, so reject.

Camera guard: the camera is excluded from both roles. This keeps the camera's
separate update path (`Camera::update`) untouched and avoids surprising
"camera-as-parent" geometry. Documented as a limitation.

### Deletion (cascade)

`Scene::deleteObject(obj)` becomes:

1. If `obj->parent_`, remove `obj` from its parent's `children_`.
2. Collect `obj` plus all descendants (DFS, deterministic order).
3. Erase every collected object from `objects_`.
4. Clear selection if it pointed at a deleted object (callers' responsibility, as today).

Cascade is the simplest, most predictable policy ("delete this thing removes the
thing and everything under it"). Reparent-on-delete (Unity editor default) is
rejected as needless complexity for this milestone (§11).

### Duplication (subtree cascade)

`Scene::duplicateObject(obj)` becomes:

1. Create root copy `obj'` (parent `nullptr`) with `obj`'s local transform and
   cloned components. Name = `obj->name() + " Copy"`.
2. For each child `C` of `obj`, recursively build a copy subtree under `obj'`
   via an internal helper `duplicateRecursive(src, newParent)` that preserves
   local transform + components + child names and calls `setParent` to attach.
3. Return `obj'`.

Child names are preserved as-is (only the root is suffixed). This means
duplicate names can exist after duplication; `findObjectByName` returns the first
match (unchanged), and the caveat is documented. Scoped name lookup is out of
scope (§13).

### `clearObjects`

Unchanged in effect: it erases `objects_` (keeping the camera). Because every
removed `GameObject` is destroyed, its `parent_`/`children_` pointers vanish with
it — no dangling references survive. No special handling needed.

## 4. World transforms

### Where the cache lives

`Transform` gains a derived world-matrix cache. The author-editable fields stay
local; the cache is written only by `Scene::updateWorldTransforms()`.

```cpp
struct Transform {
    Vec3 position = {0.0f, 0.0f, 0.0f};   // LOCAL to parent
    Vec3 rotation = {0.0f, 0.0f, 0.0f};   // LOCAL, radians
    Vec3 scale    = {1.0f, 1.0f, 1.0f};   // LOCAL

    Mat4 localMatrix() const;             // unchanged: T * Ry * Rx * Rz * S
    const Mat4& worldMatrix() const { return worldMatrix_; }

private:
    friend class Scene;
    Mat4 worldMatrix_ = identity();       // derived cache; refreshed by Scene
};
```

Default `worldMatrix_ = identity()` is safe because Scene refreshes it before any
read (see §9). For a root object that has never been refreshed, identity would be
wrong, but every read path refreshes first — covered by tests.

Rejected alternative: storing the world matrix in a parallel `Scene` map keyed by
id. Less cohesive than keeping it on the transform it derives from. Rejected.

### Computation

```cpp
// In Scene (public):
void Scene::updateWorldTransforms() {
    for (GameObject* root : objects-as-raw) {
        if (root->parent() == nullptr) {
            recomputeWorldRecursive(*root, identity());
        }
    }
}

// (private, recursive):
static void recomputeWorldRecursive(GameObject& obj, const Mat4& parentWorld) {
    Mat4 world = multiply(parentWorld, obj.transform().localMatrix());
    obj.transform().worldMatrix_ = world;        // friend access
    for (GameObject* child : obj.children()) {
        recomputeWorldRecursive(*child, world);
    }
}
```

Key points:

- **DFS from roots** guarantees parents are computed before children regardless of
  `objects_` order. This matters because **reparenting can violate vector order**
  (create A id 2, B id 3, then `setParent(A, B)` → child A has a lower id than
  parent B). The recursion makes computation order-independent.
- Every object is visited exactly once (it is a forest), so it is **O(n)**.
- `worldMatrix` = `parentWorld * localMatrix`. For a root, `parentWorld` is
  identity, so `worldMatrix == localMatrix` — **rendering of roots is unchanged**.

### Freshness contract (D9)

`updateWorldTransforms()` is called by:

1. `Scene::buildRenderCommands()` — at the start (covers editor direct edits via
   `DragFloat3`, which mutate local fields outside any Scene method).
2. `Scene::update(dt)` — at the end, after components have run (covers
   `sim.step` in automation: components mutate local rotation, then world
   matrices are recomputed before any assertion reads them).
3. `Scene::setParent()` — at the end (covers reparenting commands).
4. `DebugState::build()` — at the start (so dumps are always accurate, including
   in unit tests that build a scene and dump without rendering).
5. `assert.world_position` and any other world-reading command — defensively,
   before reading.

This guarantees every reader sees fresh world matrices without each caller having
to remember. The cost (an O(n) pass) is negligible at current scale and is far
simpler than dirty-flag invalidation, which is explicitly rejected (§11).

### Const access

`DebugState::build` currently takes `const Scene&`. To allow it to refresh the
cache, `DebugState::build` is changed to take **`Scene&`** (non-const) and calls
`scene.updateWorldTransforms()` first. Ripple:

- `editor.h`: `drawDiagnostics(...)` changes from `const Scene&` to `Scene&`
  (it already receives a non-const `Scene&` from `drawUI`).
- All other `DebugState::build` callers (`bundle.cpp`, `test_suite.cpp`,
  `command.cpp`, unit tests) already pass a non-const `Scene&` (via
  `ctx.scene` or a local), so they are unaffected.

Rejected alternative: make `worldMatrix_` `mutable` and `updateWorldTransforms`
`const` so `const Scene&` holders can refresh. This avoids the signature change
but is subtler (const method mutating state, even if only a cache). The non-const
signature change is two mechanical edits and is more explicit per AGENTS.md.

## 5. Rendering

`Scene::buildRenderCommands` changes from `localMatrix()` to `worldMatrix()`:

```cpp
void Scene::buildRenderCommands(RenderContext& ctx) const {
    // (updateWorldTransforms is called by the public buildRenderCommands wrapper;
    //  see note below on const-ness.)
    ctx.drawGround(camera_->transform().position);
    for (const auto& obj : objects_) {
        if (!obj->active()) continue;
        if (MeshRenderer* mesh = obj->getComponent<MeshRenderer>()) {
            ctx.drawCube(obj->transform().worldMatrix(), mesh->color);
        }
    }
}
```

Const-ness note: `buildRenderCommands` is currently `const`. To call
`updateWorldTransforms()` (which writes the cache) at its start, it must become
**non-const**. Its sole caller, `Application::onRender()`, holds a non-const
`Scene`, so this is a mechanical change. (This is the same trade-off as
`DebugState::build`: prefer an explicit non-const method over `mutable`.)

Effect:

- Root objects: `worldMatrix == localMatrix`, so visual output is identical.
- Children: rendered at their composed world position/orientation/scale. A
  rotating parent visibly orbits its children; a translated parent drags them.
- The ground and camera path are untouched (ground uses camera position; camera
  uses its own `viewMatrix`).

## 6. Serialization

### Format (D7)

Hierarchy is encoded structurally as a **nested `children` array**. Scenes with
no children serialize byte-for-byte as today (the `children` field is omitted
when the vector is empty), so existing scene files and tests are unaffected.

```json
{
  "objects": [
    {
      "name": "Parent",
      "position": [0.0000, 0.0000, 0.0000],
      "rotation": [0.0000, 0.0000, 0.0000],
      "scale": [1.0000, 1.0000, 1.0000],
      "components": [ {"type": "MeshRenderer", "mesh": "cube", "color": [...]} ],
      "children": [
        {
          "name": "Child",
          "position": [1.0000, 0.0000, 0.0000],
          "rotation": [0.0000, 0.0000, 0.0000],
          "scale": [0.5000, 0.5000, 0.5000],
          "components": [ {"type": "MeshRenderer", "mesh": "cube", "color": [...]} ]
        }
      ]
    }
  ]
}
```

Only **local** fields are serialized (they are the source of truth); world
matrices are derived and never serialized.

### Save

`SceneSerializer::save` recurses. For each root object (non-camera), it writes
the existing fields, then — if `children_` is non-empty — writes a `"children"`
array by recursing into each child. Children are written in `children_`
(insertion) order, which is deterministic.

### Load

The parser adds a `children` field to the per-object field switch (currently
"unknown field = error"; `children` becomes a known field). Parsing recurses:

1. Parse the root object fields (name/position/rotation/scale/components),
   `scene.createObject(name)`, set its local transform, attach components.
2. If a `children` array is present, parse each child the same way, then call
   `scene.setParent(child, parent)` to attach it. Because the parent is created
   before its children (top-down recursion), this always succeeds and produces
   correct world transforms.
3. The existing "no MeshRenderer" warning fires per object as before, recursing
   into children so invisible children also warn.

Top-down creation guarantees the parent-before-child ordering that `setParent`
and world recompute rely on.

### Why nested, not parent-by-reference

- **Name references** are fragile (duplicate names after duplication).
- **Id references** are impossible (ids are not serialized).
- **Index references** are brittle to reordering.
- **Nested structure** is self-describing, robust, diffs cleanly, and needs no
  secondary resolution pass. This is the standard approach.

## 7. Agent commands (D9 — name-based)

All new commands are **name-based** (consistent with rotator/assertion commands;
stable across load). The camera cannot be referenced as parent or child.

| Command | Behavior |
|---|---|
| `scene.set_parent <childName> <parentName>` | Attach `childName` under `parentName`. Errors: either name not found; child is camera; parent is camera; `child == parent`; cycle. Idempotent if already parented to same. Refreshes world transforms. |
| `scene.detach <name>` | Make `name` a root (parent = nullptr). Errors: not found; is camera; already a root (no-op success). |
| `scene.get_parent <name>` | Prints `Parent: <name> [<id>]` or `Parent: none`. |
| `scene.get_children <name>` | Prints each child `name [id]` on its own line, or `Children: none`. |
| `assert.parent <childName> <parentName\|none>` | Assert `childName`'s parent matches `parentName` (by name), or has no parent when `parentName` is the literal `none`. |
| `assert.world_position <name> <x> <y> <z> [tolerance]` | Assert `name`'s **world-space** position (translation of `worldMatrix`) matches within `tolerance` (default `0.01`). Refreshes world transforms first. |

Notes:

- World position is extracted from `worldMatrix`'s translation column:
  `{worldMatrix.columns[3][0], [3][1], [3][2]}`.
- `assert.world_position` is **essential** for testing hierarchy composition
  deterministically: without it, tests could only assert local position (which
  does not change when a parent moves), and would never verify that world
  transforms compose correctly.
- `assert.parent` uses the literal token `none` to mean "is a root". This is the
  one reserved token introduced; documented as such.
- Errors on parenting are **command failures** (not assertion failures) — they do
  not set `Last Assertion Failure`. Cycle/camera rejections are usage errors.
- All six commands are added to `commandTable()` so `agent.commands` /
  `agent.help` discover them automatically. No structural change to the command
  system.

### Mutation freshness

`scene.set_parent` and `scene.detach` route through `Scene::setParent`, which
refreshes world transforms. `assert.world_position` refreshes defensively. So a
script can `scene.set_parent` then immediately `assert.world_position` and see
the result, deterministically.

## 8. Editor UI

### Hierarchy panel (tree + drag-drop reparent)

`Editor::drawHierarchy` is refactored to render a **recursive tree** instead of
a flat list, while keeping the existing Create / Duplicate / Delete buttons
(which now cascade via Scene).

- Each object is an `ImGui::TreeNodeEx` (leaf flag when `children().empty()`).
  Indentation by depth is automatic. Selection on click (same `selected_` model).
- **Drag-and-drop reparenting**: an object is a `BeginDragDropSource` carrying a
  `GameObject*` payload (`"GAMEOBJECT"` type), and every node is a
  `BeginDragDropTarget`. Dropping object A onto object B calls
  `scene.setParent(A, B)` (Scene validates cycles/camera). Dropping onto the
  empty area below the list calls `scene.setParent(A, nullptr)` (detach).
- The camera renders as a non-draggable, non-drop-target leaf (it is excluded
  from hierarchy).
- A **Detach** button for the selected object calls `scene.detach(selected_)`
  (explicit alternative to drag-to-background).

`scene.duplicate`/`scene.delete` via the buttons now cascade through Scene; no
editor logic is added for recursion.

### Inspector (read-only hierarchy/world readout)

`Editor::drawInspector` gains, after the Name field:

- `Parent: <name>` (read-only `ImGui::Text`), or `Parent: (root)`.
- `World Position: <x>, <y>, <z>` (read-only `ImGui::Text`), refreshed from the
  selected object's `worldMatrix` translation.

The local `Position`/`Rotation`/`Scale` `DragFloat3` controls stay (they edit
**local** fields; the world readout updates next frame via render's
`updateWorldTransforms`). This keeps authoring local-first, matching §3/D3.

No new editor state members are required (drag-drop is stateless via the ImGui
payload). The `imgui.ini` window positions may shift on first run; that is
expected and harmless (`ImGuiCond_FirstUseEver`).

## 9. DebugState (D8)

Three additive changes. All are **substring-safe** for the existing unit tests
(which search for `[1] Camera`, `[2] Cube`, `selected: Cube [2]`,
`renderCommands: 2`, `components: MeshRenderer`, `SceneFile:`, etc.). The exact
round-trip test (`writeToFile` then `read == text`) is unaffected because it
compares whatever `build` returned against what was written.

### (a) New `Hierarchy:` tree section

Printed once, DFS from roots, indented two spaces per depth:

```text
Hierarchy:
  [1] Camera
  [2] Parent
    [3] Child
  [4] LoneRoot
```

Order: roots in `objects_` order; children in `children_` order. Deterministic.

### (b) `parent:` line in each per-object detail block

```text
  parent: Parent [2]
```
or `parent: none` for roots. Inserted alongside the existing `position:` etc.

### (c) `worldPosition:` line in each per-object detail block

```text
  worldPosition: 1.00, 0.00, 0.00
```

Translation of the object's `worldMatrix`, formatted with the existing precision.
This directly surfaces the milestone's "world transforms" in the dump, which is
the whole point for agent debugging.

`DebugState::build` calls `scene.updateWorldTransforms()` at its start (see §4
const-access note) so these values are always fresh.

## 10. Tests

### `.wbs` suite tests (in `assets/tests/`)

Note on values: `scene.create_cube` gives each object position `(0, 0.5, 0)` and
scale `(0.5, 0.5, 0.5)`. A parent's scale **does** scale its children's local
offsets (correct hierarchy behavior), so the assertions below account for the
0.5 scale. All values were verified against `math.cpp`'s `rotationY` / matrix
conventions with a standalone check during planning.

`hierarchy_parent.wbs` — static composition + parent assertion:

```text
scene.create_cube Parent            # id 2
transform.set_position 2 0 0 0      # Parent at origin (scale still 0.5)
scene.create_cube Child             # id 3
transform.set_position 3 2 0 0      # Child local (2,0,0)
scene.set_parent Child Parent
assert.parent Child Parent
# Parent scale 0.5: local (2,0,0) -> world (1,0,0).
assert.world_position Child 1 0 0
transform.set_position 2 1 2 3      # move Parent to (1,2,3)
# local (2,0,0) scaled by 0.5 -> (1,0,0) then translated by (1,2,3) -> (2,2,3).
assert.world_position Child 2 2 3
```

`hierarchy_rotate.wbs` — a rotating parent orbits its child's world position
(the headline behavior of world transforms + components):

```text
scene.create_cube Pivot            # id 2
transform.set_position 2 0 0 0     # Pivot at origin (scale still 0.5)
scene.create_cube Arm              # id 3
transform.set_position 3 1 0 0     # Arm local (1,0,0)
scene.set_parent Arm Pivot
component.add_rotator Pivot
component.set_rotator Pivot 0 90 0
# Parent scale 0.5: local (1,0,0) -> (0.5,0,0); rotationY(90°) maps that
# to (0,0,0.5) under the engine's matrix convention.
sim.step 1.0
assert.world_position Arm 0 0 0.5 0.0001
```

(Derivation verified against `math.cpp`'s `rotationY`: for θ=90°,
`columns[0] = {0,0,1,0}`, so `(0.5,0,0)` → `(0,0,0.5)`.)

`hierarchy_detach.wbs` — detaching reverts world to local:

```text
scene.create_cube Parent
transform.set_position 2 5 0 0      # Parent at (5,0,0), scale 0.5
scene.create_cube Child
transform.set_position 3 1 0 0      # Child local (1,0,0)
scene.set_parent Child Parent
# Parent scale 0.5: local (1,0,0) -> (0.5,0,0) translated by (5,0,0) -> (5.5,0,0).
assert.world_position Child 5.5 0 0
scene.detach Child
assert.parent Child none
# Detached: world == local (1,0,0).
assert.world_position Child 1 0 0
```

### Failure-path example (in `assets/tests/examples/`)

`hierarchy_cycle_fail.wbs` — exercises the cycle guard. `set_parent A B` then
`set_parent B A`; the second command fails and stops the script. Placed under
`examples/` so the suite stays green, mirroring `rotation_fail.wbs`:

```text
scene.create_cube A
scene.create_cube B
scene.set_parent A B
scene.set_parent B A
```

### Unit tests (in `src/tests/main.cpp`), mirroring existing style

- `setParent` basics: `A.parent() == B`, `B.children()` contains `A`, `A.children()` empty.
- Idempotent re-parenting to the same parent is a success no-op.
- `detach` / `setParent(child, nullptr)`: `parent() == nullptr`, removed from old parent's `children_`.
- Cycle guard: after `setParent(A, B)`, `setParent(B, A)` returns false and
  leaves the hierarchy unchanged.
- Camera guard: `setParent(camera, X)` and `setParent(X, camera)` both fail.
- `updateWorldTransforms`: parent at translate(2,0,0), child local (1,0,0) →
  child world position (3,0,0).
- `RotateComponent` on parent orbits a child's **world** position: after
  `scene.update(1.0f)` with a 90°/s parent rotator, child world position matches
  the rotated local offset (within epsilon).
- `deleteObject` cascades: deleting a parent removes it and all descendants from
  `objects()`; no dangling parent pointers remain.
- `duplicateObject` duplicates the subtree: copy has its own children with
  preserved local transforms and cloned components.
- Serialization round-trip with nested `children`: save a two-level scene, load
  into a fresh scene, verify parent/child links, local transforms, and components
  are preserved and `assert.world_position`-equivalent values match.
- Serialization backward compatibility: a flat scene (no children) saves
  identically to today (no `children` field).
- `DebugState`: `Hierarchy:` section present and correctly indented; `parent:`
  and `worldPosition:` lines present.
- Commands: `scene.set_parent`, `scene.detach`, `scene.get_parent`,
  `scene.get_children` — success + each documented error case.
- `assert.parent` (matching + `none`) and `assert.world_position` — success and
  failure (failure sets `lastAssertionFailure`).
- New commands appear in `agent.commands` and have `agent.help` entries.

These keep the `tests` executable as the source of truth and exercise every
invariant deterministically without relying on the async screenshot path.

## 12. Risks and edge cases

1. **World-matrix freshness in automation.** Highest risk for test determinism.
   Mitigated by the freshness contract (§4/D9): `sim.step` recomputes at the end
   of `Scene::update`; `setParent`/`detach` recompute; `assert.world_position`
   and `DebugState::build` recompute defensively. Covered by `hierarchy_rotate.wbs`.
2. **Reparenting violates `objects_` topological order.** A child can have a
   lower id than its parent (create A=2, B=3, `setParent(A,B)`). Mitigated:
   `updateWorldTransforms` uses DFS-from-roots, never vector order. Covered by a
   unit test for exactly this case.
3. **Cycles.** `setParent` rejects parenting an ancestor under its descendant.
   Covered by `hierarchy_cycle_fail.wbs` and a unit test.
4. **Dangling pointers.** Impossible by construction: all mutations go through
   Scene, `deleteObject` removes the whole subtree and patches the parent's
   `children_`, `clearObjects` erases everything. No raw pointer is ever freed
   while another object still references it.
5. **Camera in hierarchy.** Excluded from both roles. Documented; tested.
6. **Serialization backward compatibility.** No-children scenes serialize
   identically (the `children` field is omitted). Existing scene files load
   unchanged. Covered by the round-trip and backward-compat unit tests.
7. **Duplicate names after subtree duplication.** Children keep their names;
   `findObjectByName` returns the first match. Documented caveat; agents must use
   unique names when scripting (already required today for reliable lookups).
8. **`transform.set_position` semantics.** Still sets the **local** field. For
   roots local == world, so all existing position tests are unchanged. Documented.
9. **`DebugState::build` and `buildRenderCommands` becoming non-const.** Both
   now refresh the cache. Mechanical signature changes; all callers already hold
   non-const `Scene`. No behavior change beyond always-fresh world matrices.
10. **Editor drag-drop reparenting.** Dragging the camera or onto the camera is a
    no-op (Scene rejects). Dragging onto a descendant is rejected by the cycle
    guard. The UI simply reflects the Scene's accept/reject; no editor-side
    validation is duplicated.
11. **Orbit-test matrix derivation.** The `hierarchy_rotate.wbs` expected world
    position `(0, 0, 0.5)` was verified against `math.cpp`'s `rotationY` (for
    θ=90°, `columns[0] = {0,0,1,0}`), accounting for the parent's 0.5 scale.
    The tolerance (`0.0001`) absorbs float error. Because the test is a precise,
    deterministic check, any mistake in the derivation fails loudly during
    implementation and is corrected rather than masked.
12. **No new translation units.** All edits are to existing files, so
    `premake5.lua` (sandbox globs, tests explicit list) needs **no changes**;
    `./build.sh` / `./test.sh` regenerate identical ninja files. No build risk.

## 13. Alternatives considered and rejected

- **Tree ownership (GameObject owns children via `unique_ptr`).** Changes the
  lifetime model, breaks single-owner Scene invariant, complicates
  `findObjectById`/delete/duplicate, reorders iteration. Rejected in favor of
  flat storage + non-owning pointers.
- **Dirty-flag world-matrix caching.** Adds invalidation tracking and a class of
  "stale matrix" bugs. Eager O(n) recompute is trivially correct at current
  scale and matches the codebase's explicit style. Rejected.
- **Parent-by-reference serialization (name/id/index).** Fragile (name
  collisions), impossible (ids not serialized), or brittle (index reordering).
  Rejected in favor of nested `children`.
- **World-space authoring/editing by default.** Composes poorly and would change
  the semantics of every existing transform command/assertion. Rejected; local
  authoring with derived world is the standard, consistent model.
- **`mutable` world cache + `const` updater.** Avoids two signature edits but is
  subtler (const method mutating state). Rejected in favor of explicit non-const
  `updateWorldTransforms` + non-const `build`/`buildRenderCommands`.
- **Reparent-on-delete (children move to grandparent/root).** More complex and
  more surprising than cascade. Rejected; cascade is simplest and most explicit.
- **`scene.create_cube <name> <parent>` convenience overload.** Convenient but
  scope creep; `scene.set_parent` after creation is sufficient and more explicit.
  Rejected for this milestone.
- **Scoped name lookup (`Parent/Child`).** Would solve duplicate-name ambiguity
  but is a separate feature. Deferred (§14).
- **ECS / Transform-as-Component.** Forbidden by constraints. Rejected.
