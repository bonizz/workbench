# Milestone 0.10 Plan — Selection Highlight

Status: APPROVED. Implemented as specified below, with one approved tweak: the
overlay uses **partial alpha** (translucent yellow) so the base object stays
readable rather than being fully hidden. This requires enabling alpha blending
on the existing mesh pipeline — a single, safe flag flip (opaque meshes use
alpha 1.0, which is a no-op under src-alpha / one-minus-src-alpha blending).

## Goal

Make the currently selected object visibly highlighted in the viewport. After
picking (Milestone 0.9) the Hierarchy and Inspector update, but there is no
visual feedback in the 3D view. This milestone draws a bright, scaled-up yellow
overlay of the selected `MeshRenderer` object so it is obvious which object is
selected. The overlay respects world transforms and hierarchy, works for cube,
sphere, and plane, and clears when selection clears.

## Non-goals (explicit)

- **No transform gizmos** or drag manipulation.
- **No outline post-process** (no edge detection, no second render target).
- **No stencil buffer.**
- **No selection handles.**
- **No wireframe pipeline** unless trivial — the scale-up overlay is preferred.
- **No new component type.** Highlight is derived from existing selection.
- **No serialization changes.** Selection is runtime-only editor state.
- **No screenshot validation.** Manual visual verification only, per policy.

---

## 1. Architecture review (what exists today)

Relevant facts that constrain this design:

- **Selection state lives in `Editor::selected_`** (`editor/editor.h`).
  `Editor::selected()` returns `const GameObject*`; `Editor::setSelected()` is
  the single mutator, already called by viewport picking
  (`application.cpp:402`) and the Hierarchy.
- **`Scene::buildRenderCommands(RenderContext&)`** (`scene/scene.cpp:139`)
  refreshes world transforms, emits the ground command, then iterates
  `objects_`, skipping inactive ones, and emits one `drawShape` per active
  `MeshRenderer` using `obj->transform().worldMatrix()` and `mesh->color`.
- **`RenderCommand`** (`renderer/render_command.h`) carries `type`,
  `transform`, `color`, and ground-only grid fields.
- **`RenderContext::drawShape(shape, transform, color)`**
  (`renderer/render_context.cpp:17`) pushes a `RenderCommand`.
- **The Metal draw loop** (`metal_renderer.mm:289`) iterates `rc.commands()`,
  and for cube/sphere/plane binds the mesh pipeline and draws indexed triangles
  using `cmd.transform` and `cmd.color`. The ground is a separate case.
- **`Application::onRender`** (`application.cpp:320`) builds the
  `RenderContext` and calls `scene_->buildRenderCommands(ctx)`. It owns the
  `Editor`, so `editor_->selected()` is reachable here.
- **The camera is inactive** (`setActive(false)`), so it is already skipped by
  the `buildRenderCommands` active-object loop and has no `MeshRenderer`.
- **Primitives are unit-sized in local space** (cube `[-1,1]³`, sphere radius 1,
  plane `x,z ∈ [-1,1]`), scaled to world via `worldMatrix()`.
- **`core/math.h`** provides `scale(x,y,z)` and `multiply(left, right)`.
- **Render-command tests** live in `src/tests/test_renderer.cpp`; one calls
  `scene.buildRenderCommands(ctx)` and asserts exact command count and order.

## 2. Major design decisions

| # | Decision | Choice |
|---|---|---|
| D1 | Highlight technique | Scale-up translucent overlay: draw the selected mesh a second time, world transform scaled by 1.04, constant yellow at ~0.5 alpha so the base object reads through. |
| D1b | Blending | Enable alpha blending on the existing `meshPipeline` (flip its `blend` arg `false → true`). Safe for opaque meshes (alpha 1.0 = no-op). No new pipeline, no extra depth state. |
| D2 | How it reaches the renderer | As an ordinary extra `RenderCommand` (same `ShapeType`). The existing mesh draw loop renders it unchanged — **zero renderer draw-path edits.** |
| D3 | `RenderCommand::highlight` flag | Add `bool highlight = false` purely for inspection/tests. The renderer ignores it (draws normally). |
| D4 | Selection plumbing | Add `const GameObject* selected = nullptr` parameter to `buildRenderCommands`; `Application` passes `editor_->selected()`. Default keeps existing callers/tests compiling. |
| D5 | Where highlight is emitted | Inside the existing active-object `MeshRenderer` branch, guarded by `obj.get() == selected`. This structurally excludes nullptr / camera / inactive / no-MeshRenderer. |
| D6 | Scale anchor | `multiply(worldMatrix, scale(1.04))` — scales in local space about the object origin, so the overlay stays centered on the object under any parent transform. |

### Why scale-up overlay instead of wireframe

A wireframe overlay needs a line-primitive pipeline (or barycentric edge shader)
and per-shape edge data — more moving parts than this milestone warrants. The
scale-up overlay reuses the existing mesh pipeline and index buffers verbatim.
The selected object becomes a slightly larger solid yellow copy of itself:
visually obvious, shape-correct, and "not perfect" as the milestone allows.

### Why a second `RenderCommand` instead of a renderer branch

The overlay is geometrically identical to a normal mesh draw, just with a
different transform and color. Emitting it as a regular command means the Metal
loop needs no changes at all. The `highlight` flag is carried only so unit tests
(and future debugging) can distinguish the overlay command; the renderer never
reads it.

### Why the emission lives inside the MeshRenderer branch

Putting the `obj.get() == selected` check inside the branch that already
guarantees "active object with a `MeshRenderer`" means every exclusion the
milestone requires falls out for free:

- `selected == nullptr` → no object matches.
- camera → inactive (loop skip) and has no `MeshRenderer`.
- inactive object → skipped by the existing `!obj->active()` guard.
- object without `MeshRenderer` → never enters the branch.

No extra guard code, no separate pass over `objects_`.

## 3. Highlight emission

In `Scene::buildRenderCommands`, after emitting the normal shape command:

```cpp
ctx.drawShape(toRenderShape(mesh->shape), obj->transform().worldMatrix(), mesh->color);
if (obj.get() == selected) {
    Mat4 highlightXform = multiply(obj->transform().worldMatrix(), scale(1.04f, 1.04f, 1.04f));
    ctx.drawShape(toRenderShape(mesh->shape), highlightXform, kHighlightColor, /*highlight=*/true);
}
```

with a file-local constant:

```cpp
constexpr simd::float4 kHighlightColor = {1.0f, 0.85f, 0.1f, 0.5f}; // translucent yellow
constexpr float kHighlightScale = 1.04f;
```

The overlay command is pushed immediately after the object's normal command, so
ordering relative to other objects is preserved and it draws on top of its own
base mesh.

### Depth behaviour

The overlay is drawn with the same depth state as everything else (normal depth
test + write). Because it is uniformly larger, its faces sit slightly in front
of the base mesh from any view angle, so it renders over the original without
z-fighting. The base object's surface shows through the translucent overlay,
keeping it readable. No depth-state change is needed.

## 4. Selection plumbing

- `Scene::buildRenderCommands` signature becomes
  `void buildRenderCommands(RenderContext& ctx, const GameObject* selected = nullptr);`
- `Application::onRender` passes the editor selection:
  `scene_->buildRenderCommands(ctx, editor_->selected());`

`editor_->selected()` returns `const GameObject*`, so `buildRenderCommands` takes
a `const GameObject*` and compares against `obj.get()`. No `const`-correctness
problems: the function only reads the selected object.

## 6. Tests

Added to `src/tests/test_renderer.cpp`:

- **Highlight emitted for selected MeshRenderer object.** Build a scene with a
  cube; call `buildRenderCommands(ctx, cube)`; assert exactly one command has
  `highlight == true`, its `type` matches the cube, and its color is the yellow
  constant.
- **No highlight for `nullptr` selection.** `buildRenderCommands(ctx, nullptr)`
  emits zero highlight commands (and the existing count/order test still holds).
- **No highlight for the camera.** Selecting the camera object emits zero
  highlight commands.
- **No highlight for an inactive object.** An inactive cube, when selected,
  produces neither a base nor a highlight command.
- **No highlight for an object without `MeshRenderer`.** A selected empty object
  emits zero highlight commands.
- **Highlight respects world transform.** A selected child under a translated
  parent yields a highlight command whose transform translation matches the
  child's world position (scaled overlay still centered on the object).

Existing `buildRenderCommands` tests stay green because the new parameter
defaults to `nullptr` (no highlight commands added).

### `.wbs` tests

None required. Selection is editor-only runtime state with no agent command, and
the unit tests cover emission logic deterministically.

## 8. Risks and edge cases

1. **Translucent overlay over overlapping objects.** The overlay writes depth,
   so an object drawn afterward that sits behind the overlay can be occluded by
   it where it should show through. This only matters for objects overlapping
   the selected one and is the accepted "visually obvious, not perfect"
   trade-off.
2. **Non-uniform / zero scale.** `multiply(worldMatrix, scale(1.04))` scales the
   already-scaled world matrix uniformly; a zero-scale object stays invisible
   (overlay also zero) — acceptable and non-crashing.
3. **Plane is thin.** A 4% scale-up of a flat plane is still flat; the overlay
   reads as a slightly larger yellow plane co-planar with the original. The
   front-facing offset keeps it visible. Acceptable.
4. **Selection of a deleted object.** Out of scope — picking/Hierarchy already
   manage `selected_` lifetime; `buildRenderCommands` only reads the pointer for
   identity comparison within the same frame.
