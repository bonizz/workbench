# Milestone 0.9 Plan — Object Picking

Status: APPROVED. Implemented as specified below, with `scene.pick_ray` deferred to a future milestone.

## Goal

Allow clicking an object in the viewport to select it. Left-clicking a
`MeshRenderer` object sets the editor selection; left-clicking empty space
clears it. Picking uses a simple CPU raycast against the world-transformed
primitive geometry. The camera is never pickable, and selection stays in sync
with the Hierarchy and Inspector.

## Non-goals (explicit)

- **No GPU picking.** CPU raycast only.
- **No gizmos or drag manipulation.** Selection only.
- **No physics system, BVH, or spatial acceleration.** Brute-force over objects.
- **No multi-select, box select, or lasso.** Single left-click only.
- **No new component type.** Picking is a `Scene` operation over existing
  `MeshRenderer` objects.
- **No screenshot validation.** Manual visual verification only, per policy.

---

## 1. Architecture review (what exists today)

Relevant facts that constrain this design:

- **Selection state lives in `Editor::selected_`.** `Editor::setSelected()` is
  the single mutator; `drawHierarchyNode` and `drawInspector` already read it.
- **`Camera` has `viewMatrix()` and `projectionMatrix()`.** Projection is a
  fixed 60° vertical FOV reversed-Z perspective built in `Camera::setAspect`.
- **Renderer viewport matches the window client area.** `Window::width()` /
  `height()` report view coordinates; AppKit mouse events arrive in the same
  view coordinate space (origin bottom-left).
- **Mouse input today is only right-drag + scroll.** `Window` forwards
  `onMouseDrag` and `onScroll` to `Application`; there is no left-click path.
- **World transforms are derived and refreshed by `Scene::updateWorldTransforms`.**
  `buildRenderCommands` and `DebugState::build` call it so matrices are fresh.
- **Primitives are unit-sized in local space.** Cube `[-1,1]³`, sphere radius 1,
  plane `y=0` with `x,z ∈ [-1,1]`. `Scene::buildRenderCommands` scales them via
  `worldMatrix()`.
- **`MeshRenderer` carries the shape enum.** `MeshShape { Cube, Sphere, Plane }`.
- **Tests target lists files explicitly.** Any new `.cpp` in `tests` must be
  added to `premake5.lua`.

## 2. Major design decisions

| # | Decision | Choice |
|---|---|---|
| D1 | Ray generation | Free function `makeCameraRay` using camera basis extracted from `viewMatrix`; avoids full matrix inversion and avoids duplicating camera yaw/pitch math. |
| D2 | Intersection space | Inverse-transform the world ray into each object's **local** space, test the unit primitive, then compare hit distances in **world** space. |
| D3 | Intersection primitives | CPU slab/AABB for cube, analytic sphere for sphere, bounded plane for plane. |
| D4 | Selection behavior | Left mouse down in the viewport selects the nearest positive-hit object; a miss clears selection. |
| D5 | ImGui passthrough | Only pick when `ImGui::GetIO().WantCaptureMouse` is false, matching existing input conventions. |
| D6 | Pickable set | Active `MeshRenderer` objects only; camera and inactive objects are skipped. |
| D7 | Testability | Pure ray/intersection geometry is unit-tested; `Scene::pickObject` and `scene.pick_ray` are also tested. |

### Why local-space intersection

Testing the unit primitive in local space is far simpler than world-space OBB,
ellipsoid, or oriented-quad tests. It reuses the existing primitive definitions
and handles arbitrary rotation/scale automatically. World-space distance is used
for the final comparison so non-uniform scale does not skew "nearest."

### Why derive the ray from `viewMatrix` instead of yaw/pitch/FOV

Two natural ways to build the ray exist:

| Approach | How it works | Pros | Cons |
|---|---|---|---|
| **A. Yaw/pitch direct** | Recompute `forward`, `right`, `up` from `Camera::yaw_`/`pitch_` and `cross({0,1,0}, forward)`. | Matches human intuition about where the camera is looking. | Requires exposing or duplicating camera orientation math; risks `cross` order/sign mistakes; must stay in sync with `Camera::viewMatrix` if rotation conventions change. |
| **B. `viewMatrix` extraction** | Read the orthonormal basis directly from `Camera::viewMatrix()` columns. | Uses the exact basis already used for rendering; no new Camera getters; no duplicated trig or cross products; guaranteed consistent with the rendered image. | Slightly less semantic unless you know the columns encode `(right, up, backward)`. |

Both produce the same ray when implemented correctly. **Approach B is chosen**
because it has fewer moving parts, avoids touching `Camera`, and is the single
source of truth for the camera's orientation.

### Why no full projection-matrix inversion

The projection matrix is reversed-Z and not needed for a direction ray. Once the
camera basis is known, only the FOV and aspect are required to scale the NDC
offsets into eye-space offsets. This avoids deriving and maintaining an explicit
inverse projection.

## 3. Ray generation

From a left-click at view-space pixel `(px, py)`:

```
ndcX = (px / width)  * 2 - 1
ndcY = (py / height) * 2 - 1
```

The camera `viewMatrix` columns are:

- column 0 = world-space `right`
- column 1 = world-space `up`
- column 2 = world-space `backward` (`eye - center`)

So world view direction is `-column2`. With `tanHalfFov = tan(fovY / 2)` and
`aspect = width / height`:

```
halfH = tanHalfFov * near
halfW = aspect * halfH
rayDirWorld = normalize(right * (ndcX * halfW) + up * (ndcY * halfH) - backward * near)
rayOriginWorld = camera.transform().position
```

The `near` factor cancels during normalization, but including it makes the
formula explicit. This is implemented in a new `core/picking.h` module as
`makeCameraRay`.

---

## 4. Intersection routines

All routines operate on a unit primitive at the origin.

### Sphere (radius 1)

Quadratic solve for `|origin + t*dir|² = 1`. Returns the nearest positive `t`.

### AABB cube (`[-1,1]³`)

Slab method over the three axes. Returns the nearest positive entry distance.

### Bounded plane (`y = 0`, `x,z ∈ [-1,1]`)

Solve `origin.y + t*dir.y = 0`. If `t` is positive and the hit point's `x,z`
lie in `[-1,1]`, it is a hit.

These live in `core/picking.cpp` alongside the `Ray` struct.

## 5. Scene picking

`Scene` gains:

```cpp
GameObject* pickObject(const Vec3& rayOrigin, const Vec3& rayDir);
```

Implementation:

1. Call `updateWorldTransforms()` so world matrices are fresh.
2. Iterate `objects_` in creation order (determinism; tie-breaking is implicit
   in iteration order, but ties are unlikely with floating-point distances).
3. Skip camera, inactive objects, and objects without `MeshRenderer`.
4. For each candidate:
   - `invWorld = inverse(object->transform().worldMatrix())`
   - `localOrigin = transformPoint(invWorld, rayOrigin)`
   - `localDir = normalize(transformVector(invWorld, rayDir))`
   - Intersect against shape (cube/sphere/plane) to get local `t`.
   - If hit, transform the local hit point back to world, compute world-space
     distance from `rayOrigin`.
5. Return the object with the smallest positive world distance; `nullptr` on a
   complete miss.

Helper functions `inverse`, `transformPoint`, and `transformVector` are added to
`core/math.cpp` because they are generally useful.

## 6. Editor / viewport integration

### Input plumbing

- `Window` (`platform/window.mm`) adds a `-mouseDown:` handler for the left
  button. It converts the event to view coordinates and forwards
  `Application::onMouseButton(0, true, x, y)`.
- `Application` gains `onMouseButton(int button, bool down, float x, float y)`.

### Picking on click

`Application::onMouseButton`:

1. Ignore if `button != 0` or `!down`.
2. Ignore if `ImGui::GetIO().WantCaptureMouse` is true.
3. Convert `(x, y)` to NDC using `Window::width()` / `height()`.
4. Build a world ray via `makeCameraRay`.
5. Call `scene_->pickObject(ray.origin, ray.direction)`.
6. `editor_->setSelected(hit)` (or `nullptr` to clear).

This reuses the existing selection state, so the Hierarchy and Inspector update
automatically on the next frame.

## 7. Optional agent command: `scene.pick_ray` (deferred)

A plain C++ command to test picking without screenshots was considered:

```
scene.pick_ray <ox oy oz> <dx dy dz>
```

It would parse origin/direction, call `Scene::pickObject`, and report the hit.
This was deferred to keep the milestone focused on viewport picking only; the
unit tests for `Scene::pickObject` already cover the logic deterministically.

## 8. Tests

### Unit tests (`src/tests/main.cpp`)

- `makeCameraRay`: center ray points down the camera's view direction; corner
  rays point outward within the frustum.
- `intersectRaySphere`: hit along -Z, miss offset beyond radius, tangent hit.
- `intersectRayAABB`: hit front face, miss outside slab, ray starting inside
  returns a positive exit distance.
- `intersectRayPlane`: hit within bounds, miss due to parallel ray, miss due to
  `x/z` out of bounds.
- `Scene::pickObject`:
  - nearest of two cubes along the same ray;
  - camera not pickable;
  - inactive object not pickable;
  - object without `MeshRenderer` not pickable;
  - miss returns `nullptr`.
- `scene.pick_ray` command: hit, miss, malformed args.
- Command discovery includes `scene.pick_ray`.

### `.wbs` tests (optional)

No new `.wbs` test is strictly required; the unit tests cover the logic
deterministically. If useful, a tiny `assets/tests/examples/pick_ray_hit.wbs`
can demonstrate the command, but it is not part of the green suite.

## 9. Risks and edge cases

1. **ImGui `WantCaptureMouse` timing.** A first click on a newly-appeared ImGui
   window may be processed as a viewport click before ImGui claims it. This is
   the same limitation as existing right-drag/scroll input and is acceptable
   for this milestone.
2. **World-matrix freshness.** `pickObject` refreshes world transforms at the
   start, so it is safe even if called outside `buildRenderCommands`.
3. **Non-uniform scale.** Comparing world-space hit distances handles this
   correctly; local `t` alone would not.
4. **Degenerate inverse transform.** If an object has zero scale, `inverse`
   will produce infinities/NaNs and the candidate is simply skipped (no hit).
5. **Plane two-sidedness.** The renderer culls nothing, so the plane is visible
   from both sides. The bounded-plane test checks `y=0` regardless of ray
   direction; this is consistent with the visual plane.
6. **Ties.** Two objects at exactly the same world distance are resolved by
   iteration order (creation order). This is deterministic and rare.
