# 0.14 — Scene Persistence

Status: **steps 1‑6 implemented; step 7 implemented without Open/Save As modals.**

* `scene_document.h`, environment ownership move, nlohmann serializer, settings split,
  `lastScene` persistence, startup loading, and tests are in commit `518fa1c`.
* Editor Scene menu (New/Save), dirty tracking, and window title are in commit `422844d`.
* Open/Save As modals are intentionally deferred.

Design for a simple-but-extensible scene persistence system: JSON scene files
that capture objects + camera + sun/sky + scene settings, cleanly separated from
app settings, with robust defaults, versioning, and round-trip tests.

## 0. What already exists (reuse, don't rebuild)

- `SceneSerializer::save/load` (`scene/scene_serializer.{h,cpp}`) already
  round-trips objects, nested `children`, and components (`MeshRenderer`,
  `RotateComponent`) to a JSON-ish file via a hand-rolled tokenizer. Tests live
  in `tests/test_serialization.cpp`.
- Renderer (`metal_renderer`) has **no** file-IO knowledge already. ✅ Keep it.
- `Settings` (`core/settings.{h,cpp}`) writes `settings.json` via vendored
  `nlohmann::json` and holds window size/pos, editor panel visibility, **camera**,
  and **lighting (light+sky)**.
- `Scene` already carries `loadedScenePath_`.

### Gaps this milestone closes
1. **Camera + sun/sky are app settings, not scene data.** They live on
   `Application` (`lightSettings_`, `skySettings_`) and in `settings.json`. A
   scene file currently captures only objects.
2. No top-level **version** field.
3. No **"load last scene on startup"**.
4. Two JSON styles (custom tokenizer vs. nlohmann). Unify on nlohmann.
5. No editor **New/Open/Save/Save As** UI (only agent commands).

## 1. Data model structs

A scene file is a single document. Introduce a plain in-memory snapshot type so
serialization is a pure `Scene <-> SceneDocument <-> JSON` mapping and the
serializer never reaches into renderer or app state.

```cpp
// scene/scene_document.h  (POD-ish, no behavior, no IO)
struct SceneEnvironment {        // "sun/sky"
    LightSettings light;         // reuse renderer/render_types.h
    SkySettings   sky;           // reuse existing struct
};

struct SceneCameraState {
    Vec3  position{0, 3, 5};
    Vec3  rotation{0, 0, 0};     // radians
    float moveSpeed = kDefaultMoveSpeed;
    // (fov/near/far if/when Camera exposes them)
};

struct SceneSettings {           // scene-relevant, non-object knobs
    // start empty; grow here (e.g. grid on/off, gravity) without touching
    // app settings or the object model.
};

struct SceneDocument {
    int               version = kSceneVersion;   // currently 1
    SceneCameraState  camera;
    SceneEnvironment  environment;
    SceneSettings     settings;
    // objects live in Scene; the serializer walks Scene directly for the
    // object array (already implemented). The struct above covers the
    // non-object header.
};
```

`LightSettings`/`SkySettings` already exist in `renderer/render_types.h` — reuse
them; do not redefine. `kSceneVersion` and `kDefaultMoveSpeed` are constants in
`scene_document.h`.

### Runtime ownership of environment
Today `lightSettings_`/`skySettings_` live on `Application`. To let the scene
"capture sun/sky" cleanly, **move them onto `Scene`** as a `SceneEnvironment
environment()` accessor. `Scene` becomes the single in-memory source of truth for
everything scene-relevant (objects, camera, environment, settings). The serializer
then needs only a `Scene&`.

## 2. File layout

```
assets/scenes/<name>.scene.json   # scene documents (versioned)
assets/scenes/default.scene.json  # shipped starter scene (the current hard-coded cube)
settings.json                     # APP settings only (project root, as today)
```

Scene document shape (v1):

```json
{
  "version": 1,
  "camera":   { "position": [0,3,5], "rotation": [0,0,0], "moveSpeed": 5.0 },
  "environment": {
    "light": { "direction": [-0.5,-1,-0.75], "ambient": 0.15, "diffuse": 1.0 },
    "sky":   { "...": "existing SkySettings fields" }
  },
  "settings": {},
  "objects": [ { "name": "Cube", "position": [...], "components": [...] } ]
}
```

`settings.json` after the split keeps **only** app-level state:

```json
{
  "window":  { "width": 800, "height": 600, "x": 0, "y": 0 },
  "editor":  { "show_inspector": true, "...": "panel visibility" },
  "lastScene": "assets/scenes/default.scene.json"
}
```

`camera` and `lighting` are **removed** from `settings.json` (they move into the
scene). `lastScene` is added.

## 3. Ownership boundaries

| Concern | Owner | Notes |
|---|---|---|
| Runtime objects, camera, environment, scene settings | `Scene` | Single source of truth in memory. |
| `Scene <-> JSON` mapping | `SceneSerializer` | Only place that knows the file format. Knows `Scene` + `nlohmann::json` + `std::filesystem`. |
| Reading/writing app settings file | `Settings` | Window, panels, `lastScene`. No scene data. |
| Orchestration (what to load when, dirty tracking, paths) | `Application` (+ a small `SceneIO` helper) | Decides startup load, wires save/load to UI. |
| Drawing | renderer | **Never** touches files; reads `scene.environment()` via `RenderContext`. Unchanged. |
| Editor UI | `Editor` | Emits intents (new/open/save/saveAs); calls back into `Application`. No direct fs access beyond listing scene files for the picker. |

Renderer-must-not-know-IO is already satisfied and stays that way: the render
path keeps reading light/sky from a struct; only its *origin* moves from
`Application` to `Scene`.

## 4. Startup / load / save flow

**Startup**
1. `Settings::load*` → window size/pos, editor panels, `lastScene` path.
2. Construct `Scene` with **defaults** (default camera, default environment,
   empty objects).
3. If `lastScene` is set and the file loads → populate scene from it.
   Else load `assets/scenes/default.scene.json` if present; else keep the
   built-in default scene (today's hard-coded cube as the fallback).
4. Apply camera-from-scene only in `liveSimulation_` (preserve current
   automation-determinism behavior).

**Load (Open / agent `scene.load`)**
1. `SceneSerializer::load(scene, path, error, &warning)` replaces objects,
   camera, environment from the document; missing fields → defaults.
2. On success: set `loadedScenePath_`, clear dirty flag, set selection to none,
   update `Settings::setLastScene(path)`.
3. On failure: keep current scene, surface `error` in the console/UI.

**Save (Save / Save As / agent `scene.save`)**
1. Serialize `Scene` → document → JSON at the target path (`create_directories`).
2. Update `loadedScenePath_`, clear dirty, update `lastScene`.

**Shutdown**
- App settings always saved.
- Scene: **do not silently overwrite** the user's file. Either (a) save only on
  explicit Save, or (b) autosave to a separate `assets/scenes/_autosave.scene.json`
  and offer it on next launch. Recommend (a) for v1 + a dirty-flag prompt later.

## 5. Editor UI changes (imgui)

Add a **Scene** menu / toolbar (and the user explicitly asked for imgui load/save):

- **New** — reset to default scene (prompt if dirty).
- **Open…** — modal listing `assets/scenes/*.scene.json` (imgui has no native
  file dialog; use a simple `Selectable` list built from a `std::filesystem`
  directory scan + a manual filename input). Selecting calls `Application::loadScene`.
- **Save** — `Ctrl+S`; writes to `loadedScenePath_`, or falls back to Save As if
  unsaved.
- **Save As…** — modal with a filename text field; writes to
  `assets/scenes/<name>.scene.json`.
- **Window title / status**: show scene name + `*` when dirty.
- **Dirty tracking**: flip a `sceneDirty_` flag on object/camera/env edits
  (coarse is fine: set dirty on any inspector/gizmo/hierarchy mutation).

Editor stays UI-only: it calls `Application` methods; it does not serialize. The
only fs it touches is enumerating scene files for the Open list (acceptable, or
route that through a `SceneIO::listScenes()` helper to keep it clean).

## 6. Implementation steps (small commits)

1. **`scene_document.h` + constants.** Add `SceneDocument`, `SceneEnvironment`,
   `SceneCameraState`, `SceneSettings`, `kSceneVersion`, `kDefaultMoveSpeed`.
   No behavior. (Pure addition, no wiring.)
2. **Move environment onto `Scene`.** Add `SceneEnvironment environment_` +
   accessor; relocate `lightSettings_`/`skySettings_` reads in `application.cpp`
   render path and editor wiring to `scene_->environment()`. Behavior identical.
   (Touches render path — verify visually.)
3. **Migrate `SceneSerializer` to `nlohmann::json`.** Replace the custom
   tokenizer body with nlohmann (already vendored, used by `Settings`). Keep the
   public `save/load` signatures. Add top-level `version`, `camera`,
   `environment`, `settings`. Use `.value(key, default)` for every field so
   missing → default. Existing object/children/component mapping preserved.
4. **Camera + environment round-trip in serializer.** Wire `Scene::camera()` and
   `Scene::environment()` into save/load.
5. **Split `Settings`.** Remove camera + lighting from `settings.json`; add
   `lastScene` get/set. Update `Application::saveSettings`/`init` to stop
   read/writing camera+lighting via `Settings`.
6. **Startup load.** Implement the §4 startup sequence; ship
   `assets/scenes/default.scene.json` (export current hard-coded cube once).
7. **Editor UI.** Scene menu, Open/Save/Save As modals, dirty flag, title.
8. **Docs.** Update `docs/scenes.md` (new format, version, env/camera capture,
   app-vs-scene split, UI workflow).

Steps 1–2 and 3–4 can each be one commit; keep 5 and 6 separate so the settings
split is bisectable.

## 7. Risks & simplifications

- **Two JSON styles → unify on nlohmann (one-time format change).** The on-disk
  byte layout of existing `.scene` files changes once. Acceptable; tests pin the
  new shape. Alternative (keep custom parser, bolt a header on) is uglier and
  re-implements `.value()` defaults by hand — not recommended.
- **Moving light/sky into `Scene` touches the render path** (`application.cpp`
  ~L66–74, L335–337). Low logic risk, but verify lighting still looks right
  (run the app, not just tests).
- **Camera duplicated in two places today.** Pick **scene** as the source of
  truth; delete the `Settings` camera path to avoid drift. Automation-mode
  camera behavior (`liveSimulation_`) must be preserved.
- **No native file dialog in imgui.** Simplify: directory-scan list + filename
  text field. No new dependency.
- **Shutdown autosave.** Don't overwrite the user's file silently — explicit Save
  for v1; dirty-prompt/autosave-slot is a later add.
- **ObjectIds remain non-serialized** (already the contract) — ids reassigned on
  load; name-based references stay stable.
- **Versioning is forward-only stub.** v1 only; loader reads `version`, warns on
  unknown-higher, best-effort parses. Add a migration switch when v2 lands.

## 8. Tests (`tests/test_serialization.cpp`)

Extend existing cases:
- Round-trip **camera** (position/rotation/moveSpeed) survives save→load.
- Round-trip **environment** (light direction/ambient/diffuse + sky fields).
- **Missing `version`** → defaults to 1, loads.
- **Missing `camera`/`environment`/`settings`** blocks → defaults applied, load
  succeeds (the "robust defaults" requirement).
- **Unknown higher `version`** → load succeeds with warning (best-effort).
- **App/scene separation**: loading a scene does not mutate window/editor app
  settings (assert `settings.json` untouched).
- Keep existing object/children/missing-mesh/unknown-mesh cases.
