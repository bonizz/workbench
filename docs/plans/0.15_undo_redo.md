# Plan: v1 editor undo/redo (snapshot-based)

File: `docs/plans/0.15_undo_redo.md`

Status: IMPLEMENTED.

## Goal

Add undo/redo for major scene-authoring edits, using **snapshot-based** undo
(serialized Scene snapshots in memory), not command-based undo.

## Scope

Undoable (push a checkpoint **before** the edit):

- Object create / delete / duplicate / detach / reparent (drag-drop + buttons).
- Inspector: transform edits (Position/Rotation/Scale), mesh shape, material
  color, RotateComponent velocity, add component.
- Lighting panel: light direction/ambient/diffuse, sky colors, sun
  radius/intensity, presets.

**Excluded** (no checkpoint):

- Normal camera navigation + Reset Camera (camera is never restored by undo).
- Selection changes, panel layout, open/save.
- Object rename (inspector Name field) — not in the v1 scope list and text
  input would need separate coalescing.
- Agent-command edits (`scene.create_primitive` etc.) — separate surface; they
  bypass the editor. Could route through a shared history later.

## Model

`SceneHistory` (in `scene/`, no ImGui) holds two `std::deque<std::string>`
(undo/redo) of compact JSON snapshots from `SceneSerializer::serialize`.

- `push(scene)`: snapshot the current scene → undo stack; clear redo; trim to
  `kMaxEntries == 50` (oldest first).
- `undo(scene)`: snapshot live → redo; pop undo → restore.
- `redo(scene)`: snapshot live → undo (trim); pop redo → restore.

Each push records the **pre-edit** state, so undo restores the state before the
edit. `clear()` wipes both stacks (used on scene load/new).

## Snapshot serialization

`SceneSerializer::serialize(scene)` → `buildDocument(scene).dump()` (compact).
`SceneSerializer::deserialize(scene, text, error)` applies environment + objects
but **deliberately skips the camera** — navigation must stay live across undo.
`save`/`load` were refactored to share `buildDocument`/`applyDocument`.

## Coalescing continuous ImGui edits

Sliders/color editors fire every frame while active. To avoid hundreds of
undo entries per drag, continuous widgets push exactly once via
`Editor::pushUndoOnActivate(scene)`, which snapshots on
`ImGui::IsItemActivated()` (the frame the drag begins). Discrete actions
(buttons, menus, combo, drag-drop) push immediately before mutating.

The pending-snapshot lifecycle lives on `SceneHistory`
(`beginPending` / `commitPending` / `discardPending`) so it is unit-testable and
shared by two callers:

- **ImGui widgets** (`Editor::coalesceEdit`): `beginPending` on
  `IsItemActivated()`, `commitPending` on `IsItemDeactivatedAfterEdit()`, and
  `discardPending` on a bare `IsItemDeactivated()` (click without drag).
- **Viewport translate gizmo** (`Application::onLeftMouseMove`): `beginPending`
  on the first move of the drag (pre-edit, before the position is applied),
  `endDragEdit` → `commitPending` on mouse release. A click on the gizmo
  handle without moving pushes no entry (`gizmoDragMoved_` stays false).

Discrete actions (create/delete/duplicate/detach/reparent/buttons/combo/
presets) call `Editor::pushUndo(scene)` → `history_.push(scene)` immediately
before mutating.

## Keybindings

`Cmd+Z` undo, `Cmd+Shift+Z` redo. `Application::onKeyEvent` gained a
`shiftModifier` param (read from `NSEventModifierFlagShift` in `window.mm`) to
distinguish the two, since `charactersIgnoringModifiers` is case-insensitive.

## Side effects after undo/redo

`Editor::undo/redo` clear `selected_` (snapshots don't track selection, and
restored GameObjects have fresh pointers) and call `markSceneDirty()` so the
title's `*` updates.

## Scene boundaries

`Application::loadScene` (on success) and `newScene` call
`editor_->clearHistory()` so you can't undo across an open/new. Save does not
touch history (it doesn't change the scene). `init` needs no clear (history
starts empty and no push fires during the initial load).

## Tests (`src/tests/test_undo.cpp`)

1. create object → undo (empty) → redo (object back, name preserved).
2. transform edit → undo → redo (refetch object: snapshots get fresh ids).
3. lighting edit → undo → redo.
4. delete object → undo (object + components restored) → redo.
5. cap: 60 pushes → `undoCount() == 50`; 50 undos succeed, 51st fails; scene
   reflects the 10 dropped oldest checkpoints.
6. redo cleared by a new push (branching edit).
7. coalesced pending edit: several mutations between beginPending/
   commitPending collapse to one undo entry.
8. discardPending leaves history untouched.

The viewport gizmo drag itself (`Application` + window input) isn't unit-tested
(it needs the platform window/renderer); its core logic is the `SceneHistory`
pending API covered by tests 7–8, and `Editor::beginDragEdit`/`endDragEdit`
are thin forwarders.
