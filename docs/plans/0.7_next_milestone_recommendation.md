# Next Milestone Recommendation — after 0.6 (Transform Hierarchy)

Status: RECOMMENDATION (no code changes; not yet an approved plan). This is an
opinionated analysis of what to build next and what debt to clear first.

## Where the project actually is

The milestone history shows a **deep agent/test/scene spine and a frozen
renderer**:

- 0.2–0.6 built scene hierarchy, authoring, components, lifecycle, and transform
  hierarchy — plus an entire 0.4.x sub-series on the agent loop (script runner,
  capture, repro bundles, CLI automation, assertions, test suites) and 0.5.1
  richer assertions.
- The renderer has not meaningfully moved since the start. `metal_renderer.mm`
  has exactly one `cubePipeline`, one `ShapeType::Cube`, one hardcoded cube
  index buffer. `Vertex` is position + color, **no normals**.

So today you can build a parented, animated, serialized, agent-testable scene
graph of **identical colored cubes**. The composition machinery is far ahead of
what there is to compose.

## Recommendation: Milestone 0.7 — Primitive Mesh Library

**Give `MeshRenderer` real, selectable geometry.** Wire the already-existing-but-
dead `mesh` string field to actual primitives (cube + sphere + plane, maybe
cylinder), generated procedurally CPU-side exactly like the cube is today.

Sketch of the shape (not a committed design):

- Renderer holds a small map of shape → vertex/index buffer, built procedurally
  the way the cube already is. No asset files.
- `RenderContext::drawShape(shape, transform, color)` keys off an enum; the
  `RenderCommand` carries the shape.
- `component.set_mesh <name> <shape>` validates against the known shape set;
  `MeshRenderer::mesh` finally drives what is drawn.
- Serialization is unchanged in shape (the `mesh` field already round-trips); it
  just stops being ignored.

### Why this is the right next step

1. **It pays down debt that already exists.** `MeshRenderer::mesh = "cube"` is
   serialized, stored, and round-tripped — and **completely ignored by the
   renderer** (single hardcoded `cubePipeline` / `ShapeType::Cube`). The data
   model already claims something the renderer can't honor. This milestone turns
   that lie into a feature instead of leaving it to rot or deleting it.
2. **It's the highest visible-value gap.** Everything else (lighting, materials,
   picking) is more compelling once objects aren't all identical cubes.
3. **It fits the architecture and the philosophy.** `ShapeType` enum already
   exists. No asset pipeline, no registry, no new subsystem — just more
   procedural buffers and a pipeline that's already parameterized. "Build
   features because prototypes need them": a prototype needs to tell a floor
   from a ball.
4. **It's largely testable in the existing harness.** You can deterministically
   `assert.has_component`, assert the mesh/shape name, and assert the
   render-command shape count without touching the async screenshot path.

### Tradeoffs of this choice

- **Verification gap.** Metal `.mm` code is the least unit-tested area. Geometry
  *correctness* (does the sphere look like a sphere) is only checkable via the
  async capture path, not the deterministic suite. You can assert shape *type*
  deterministically, but not pixels. Accept that primitives are validated by
  capture/eye; the spine stays validated by the suite.
- **Breadth, not depth.** It doesn't advance "engine architecture
  understanding" as much as lighting would. But it *unblocks* lighting — which
  is why it is sequenced first.

### Alternatives considered and ranked below it

- **Lighting + normals (strong #2).** More architecturally educational: forces
  the `Vertex` format to grow a normal, a normal matrix in `ShaderUniforms`, and
  light params as uniforms. But (a) it requires the vertex-format surgery first,
  and (b) lighting a single cube is dull — you want varied geometry to
  appreciate it. Do geometry → then lighting as 0.8.
- **Component registry / serializer-fork refactor.** Tempting because the
  per-type fork is now smeared across serializer (save+load), `debug_state`,
  `editor`, and `command`. **Don't.** AGENTS.md explicitly says no registry until
  a third component type justifies it, and there are two. Flag it, don't build
  it.
- **Editor viewport picking / transform gizmos.** Nice, but the workflow is
  agent-first; editor mouse-manipulation is lower priority than what agents and
  the test suite can drive.

If the goal is "learn engine internals" over "make the scene look like
something," flip 0.7 and 0.8 (lighting first). Geometry leads because it clears
existing debt and is the prerequisite for lighting to be interesting.

## Technical debt to pay down first

Two cheap items that reduce friction for any command-touching milestone, and
should land before 0.7:

1. **Assert-handler duplication in `command.cpp` (1307 lines and growing).**
   `cmdAssertPosition`, `cmdAssertScale`, `cmdAssertWorldPosition`, and
   `cmdAssertRotation` are near-identical ~45-line copies: same arg-count check,
   same `parseFloat` triple, same optional-tolerance block, same `near` lambda,
   same expected/actual formatting. Collapse them into one
   `assertVec3(ctx, args, label, actualVec)` helper. Every new `assert.*` command
   is currently a copy-paste; fix this before adding more (0.7 may want
   `assert.mesh`).

2. **Consolidate the duplicated `kRadToDeg`.** Conventions §4 documents that
   `command.cpp:30` and `debug_state.cpp` each redefine a private `kRadToDeg`
   even though `core/math.h` got the shared `kRadToDeg` / `kDegToRad` in 0.5.
   That consolidation was deliberately deferred then; it is now safe and trivial.
   Small, but it is two sources of truth for a constant.

Fold into 0.7 rather than pay separately:

3. **The dead `mesh` field** — fixing it *is* the milestone, so don't touch it
   beforehand.

Flag and watch, not pay now (consistent with "optimize after measuring"):

4. **Eager O(n) `updateWorldTransforms()` on every `buildRenderCommands` and
   every `DebugState::build`.** Explicitly chosen over dirty flags in the 0.6
   plan; correct at current scale. Revisit only when a frame profile says so.
5. **The per-type serialization/debug/editor fork.** Fine at two component
   types; the documented trigger to reconsider is a third. A geometry milestone
   that adds shapes *to the existing `MeshRenderer`* (rather than a new
   component) keeps you at two — another reason to prefer that framing.

## Bottom line

Pay debt #1 and #2 (about a half-day of cleanup), then do **0.7 Primitive
Meshes**, with **Lighting queued as 0.8**.
