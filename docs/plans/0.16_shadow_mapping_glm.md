# Plan: Basic directional-light shadow mapping (0.16)

File: `docs/plans/0.16_shadow_mapping.md`

Status: PROPOSED (not yet implemented). Awaiting approval before writing code.

## Goal

Add the simplest useful directional-light shadow map to the Workbench
Metal renderer: a single shadow map, one directional light only, no
cascades, no point/spot lights.

The aim is to understand how shadow mapping fits into the existing
architecture — not to introduce a generic render graph or future-proof
abstractions. Keep the change minimal and incremental.

## Architecture fit (how it slots into the engine today)

Render flow today:

- `Application::onRender` builds a `RenderContext` (camera, light, sky,
  draw commands) and calls `MetalRenderer::draw()`.
- `MetalRenderer::draw()` is one method on one `MTLCommandBuffer`:
  clear one offscreen color+depth pass (sky → meshes → grid), blit to
  the drawable, then run the ImGui UI pass.
- Depth is **reversed-Z everywhere**: `perspective()` maps near→1.0 /
  far→0.0, the depth attachment clears to `0.0`, the depth state is
  `MTLCompareFunctionGreater` + write, format `MTLPixelFormatDepth32Float`.
- Light convention: `LightSettings.direction` is the direction light
  *travels*; surface-to-light is `-direction` (see `conventions.md` §4).
- ImGui uses `imgui_impl_metal`, which treats `id<MTLTexture>` as the
  texture handle and samples via `texture2d<float>`. A raw `Depth32Float`
  shadow map **cannot** be shown directly; it needs a depth→color debug
  copy.

Insertion strategy — mirror the existing camera/light/sky flow:

1. **Application** owns the *values*: compute a light view-projection
   (orthographic) that frames the scene.
2. **RenderContext** carries them (`setShadow(...)`), pure data.
3. **MetalRenderer** owns the *GPU resources and passes*: a shadow-map
   depth texture, a depth-only shadow pipeline, a shadow render-pass
   encoder before the existing scene pass, and a debug-copy pipeline.

Why this split: the renderer stays "dumb" (consumes a `RenderContext`),
and the scene/AABB knowledge needed to fit the light frustum lives at
the Application/Scene level. The renderer change is purely mechanical
(one more pass + one more texture bind).

## Scope

In scope:

- Shadow depth pass.
- Shadow map depth texture.
- Light view-projection matrix (ortho, scene-fitted).
- Sampling of the shadow map during the main mesh lighting pass.
- ImGui visualization of the shadow map (via a depth→color debug copy).
- Editor panel: enable toggle, depth-bias slider, debug view.

Out of scope (deliberately deferred):

- Cascaded shadow maps.
- Point lights, spot lights, multiple lights.
- Shadow-caster/receiver component split (everything the mesh pipeline
  draws casts *and* receives).
- Generic render graph / pass abstraction.
- Serializing shadow settings or exposing them via agent commands
  (would follow `conventions.md` §6/§7 additive rules; not needed for v1).
- Large-scale refactors.

### New structs / classes (minimal, none of them "frameworks")

- `Mat4 ortho(float left, right, bottom, top, nearZ, farZ)` in `math` —
  reversed-Z variant consistent with `perspective()`.
- `ShadowUniforms { simd::float4x4 lightViewProj; }` — the only
  per-draw constant the lighting shader needs beyond what it has.
- (Editor-only) POD fields for the panel: `bool shadowsEnabled`,
  `float depthBias`, `bool visualizeShadowMap`. Lives as `Editor`
  members or in `SceneEnvironment` if serialization is wanted later.

No new classes, no shadow-caster/receiver component split, no render-pass
abstraction. Casters vs. receivers is implicit: *everything drawn by the
mesh pipeline renders into the shadow map; everything drawn by the mesh
pipeline samples it.* The grid/sky use separate pipelines and are simply
skipped in both directions — matching "smallest incremental."

## Render pass ordering

Within the existing single `MTLCommandBuffer`, in this order:

1. **Shadow depth pass** (new) — render meshes (cube/sphere/plane) with
   `shadow_vertex` into `shadowMapDepth` (`Depth32Float`). Clear `0.0`,
   compare `Greater`, write enabled (**reuse the existing
   `depthStencilState`** — reversed-Z is already set up). Store.
2. **Scene pass** (existing, lightly extended) — sky (unchanged), meshes
   (now sample the shadow map), grid (unchanged).
3. **Debug copy pass** (new, optional) — fullscreen depth→color into
   `shadowDebugColor` (`R8Unorm` or `BGRA8Unorm`).
4. **Blit** scene color → drawable (existing).
5. **UI pass** — ImGui; the panel may `ImGui::Image` `shadowDebugColor`.

Why one command buffer with ordered encoders: Metal resolves the
render-target→shader-read hazard between two encoders on the same
command buffer automatically, so no explicit barrier is needed. This is
the cheapest correct way to feed the shadow map into the same frame.

## Metal-specific pitfalls

1. **Texture usage flags.** The shadow map needs both
   `MTLTextureUsageRenderTarget` (depth attachment) **and**
   `MTLTextureUsageShaderRead` (sampled next pass). Miss the second and
   sampling silently returns nothing or validation asserts.
2. **Pixel format of the shadow map.** `MTLPixelFormatDepth32Float`
   (what the engine already uses). A depth attachment *requires* a
   depth-format texture; depth cannot be rendered into `R32Float`.
3. **StoreAction must be `Store`.** The shadow pass instinct is
   "depth-only, throw it away" (`DontCare`) — that destroys the data
   before the scene pass samples it. Must `Store`.
4. **Sampling a depth texture.** Bind it as `depth2d<float>` and use
   `setFragmentTexture:`. A `texture2d<float>` binding will not accept a
   depth-format texture.
5. **Sampler compare function.** If using hardware depth-compare
   (cheap PCF), `compare_function` must be non-`none` and the texture
   must be depth format, or Metal validation errors. For v1 use a
   **manual** compare (recommended): keep `compare_function = none` and
   the sampler returns the raw depth value.
6. **Encoder order is mandatory.** Shadow encoder must `endEncoding`
   before the scene encoder begins; reordering silently breaks the
   dependency tracking. No cross-encoder barrier work is needed beyond
   correct ordering.
7. **Depth-only pipeline has no color attachment.**
   `colorAttachments[0].pixelFormat = MTLPixelFormatInvalid`,
   `depthAttachmentPixelFormat = MTLPixelFormatDepth32Float`. A fragment
   function is optional; omit it and set `fragmentFunction = nil`.
8. **ImGui can't show depth directly.** `imgui_impl_metal` samples user
   textures as `texture2d<float>`; it cannot bind `depth2d`. Hence the
   explicit debug-copy pass.
9. **`framebufferOnly` is already `NO`** on the layer (set for the
   screenshot readback), so offscreen-texture flows like this are already
   permitted — no change needed, but worth knowing why it "just works."
10. **Fixed shadow-map size is independent of the window.** Allocate it
    once (e.g. 2048²) in the constructor, **not** in `resize()`. Tying it
    to `resize()` needlessly recreates it on every window resize.

## Reversed-Z pitfalls (engine is reversed-Z everywhere)

This is the single most likely source of "everything is shadow" /
"nothing is shadow" bugs. The engine maps near→depth 1.0, far→0.0,
clears to `0.0`, compares `Greater`: **larger depth value = closer to
the camera/light**.

- **The shadow test is inverted versus the textbook.** Textbooks assume
  near→0/far→1 and write "lit if `fragmentDepth ≤ storedDepth`." Under
  reversed-Z the stored depth is the *largest* (nearest) value, so the
  rule becomes **lit if `fragmentDepth ≥ storedDepth`** (within bias).
  Flip the comparison and you get full-black or full-light.
- **Bias direction also flips.** Hardware `setDepthBias` sign
  conventions are tuned for standard-Z. For v1, use a **shader-side
  epsilon** applied to the *reference* depth, which is unambiguous:
  `lit = (fragmentDepth ≥ storedDepth − ε)`. Removes a whole class of
  "which way does bias go" confusion.
- **Clear value is `0.0` (far), not `1.0`.** Reusing the existing
  `depthStencilState` (clear `0.0`, `Greater`, write) is correct for
  the shadow pass — don't make a "normal-Z" shadow map; mixing
  conventions is worse than staying consistent.
- **Hardware-compare samplers must also flip.** If switching to
  sampler-side PCF later, the compare function for reversed-Z is
  `greater_equal` (lit direction), *not* the usual `less_equal`.
- **Ortho projection must be reversed-Z too.** `ortho()` must map
  near→1.0 / far→0.0, exactly like `perspective()`. A standard
  (near→0) ortho here would produce a depth range that disagrees with
  the `Greater` compare and self-inverts.
- **Bias is still needed even with `Depth32Float`.** Acne comes from
  *discretization in the shadow map's XY resolution and depth
  quantization at grazing angles*, not from z-precision alone. Bias is
  not optional.

## Recommended initial shadow map resolution

**2048 × 2048.** The standard quality/memory sweet spot (16 MB of
`Depth32Float`), enough that a single cube-on-a-plane test shows a
crisp edge, and leaves headroom to add bias/PCF without the map being
the limiting factor. Drop to **1024²** only if you want to make
aliasing deliberately visible for debugging. Don't go 4096² yet — it
just hides correctness bugs.

## Recommended initial depth-bias values

Two layers, applied together (both small):

- **Back-face / face-culling trick (do this first, halves the bias
  needed):** render the shadow pass with back-face culling so only the
  *far* faces of occluders are written into the map. Eliminates the
  front-face self-shadowing that causes most acne. Requires confirming
  triangle winding in `mesh_geometry.cpp`; the current scene pass draws
  **without** culling, so this is a *new* cull-mode setting on the
  shadow encoder only. *Fallback if winding is unclear:* skip culling
  for v1 and lean on the epsilon.
- **Shader-side epsilon** (reversed-Z-safe): start `ε ≈ 0.001` in
  normalized light clip-space [0,1]. Tune up to ~0.003 if acne persists,
  down to ~0.0003 if peter-panning (shadows detaching from their caster
  bases) appears.
- **Optional hardware bias** (only if the epsilon alone isn't enough):
  `setDepthBias:constant:slopeScale:clamp:` with `slopeScale ≈ 1.5`,
  `constant ≈ 0.0`, `clamp ≈ 0.0` to start. Because of the reversed-Z
  sign trap, prefer `slopeScale` (sign-stable) over a large `constant`.

Expect to tune these on the very first lit frame; treat these as "where
to put the sliders first," not as final.

## Incremental checkpoints

Each checkpoint is independently verifiable and leaves the build green.

### Checkpoint 0 — Math primitives (no rendering change)
- **Compiles:** `ortho()` and the light-view builder; a unit test
  asserting `ortho()` maps the near plane to depth 1.0 and far to 0.0,
  and that the light `lookAt` looks along `+direction`.
- **Renders:** Identical to today.
- **Verify:** `./test.sh` (unit tests pass); `ortho()` corner-mapping
  test. Also test the **degenerate-up case** (`direction = {0,-1,0}`,
  sun straight overhead) — plain `lookAt(up={0,1,0})` goes singular
  here, so this is where the "pick an alternate up vector" guard goes.
  Verify by assertion that the produced basis is orthonormal for that
  input.

### Checkpoint 1 — Empty shadow pass (clear only)
- **Compiles:** shadow-map texture creation, a depth-only shadow
  pipeline, and a second render-pass encoder in `draw()` that does
  nothing but `Clear`/`Store` on the shadow map (no draws yet).
- **Renders:** Identical to today (the shadow map is allocated and
  cleared but unused).
- **Verify:** No Metal validation errors (the `usage` flags, store
  action, and encoder ordering are all correct even though nothing is
  drawn). `./build.sh` and a manual launch; the existing scene is
  pixel-identical. This checkpoint exists purely to de-risk the
  *plumbing* (texture flags, encoder order, store action) before any
  shader math.

### Checkpoint 2 — Render casters into the shadow map (still no sampling)
- **Compiles:** `shadow_vertex` added; meshes drawn into the shadow pass
  using `lightViewProj`; the debug-copy pipeline; `Editor` shows the
  debug texture in an ImGui image.
- **Renders:** The 3D scene is still identical, **but the ImGui "Shadow
  Map" panel now shows the depth silhouette of the cube from the
  light's point of view**.
- **Verify:** Rotate the light direction in the existing Lighting
  panel; the silhouette in the debug panel should move/rotate
  accordingly. Put the cube near the edge of the ortho frustum and
  confirm it stays visible. This is the "we can actually see the shadow
  map" milestone, and it isolates the shadow pass from the sampling
  math, so if it's wrong you know it's the frustum/matrix, not the
  compare.

### Checkpoint 3 — Sample in the main pass (hard 0/1 shadow)
- **Compiles:** `fragment_main` samples the shadow map, computes the
  reversed-Z compare (`lit = fragmentDepth ≥ storedDepth − ε`),
  multiplies the diffuse term by the factor.
- **Renders:** A hard-edged shadow of the cube cast onto a `Plane`
  receiver. Add a `Plane` + a `Cube` to the scene if not present.
- **Verify:** Shadow appears on the plane, on the side *away from*
  `-direction` (i.e., away from the visible sun, per the engine's light
  sign convention). Move the light → shadow swings the right way.
  *Expect acne and/or peter-panning at this stage* — that's what CP4
  fixes. If everything is black or everything is lit, it's the
  reversed-Z compare flip (above) — recheck the inequality direction
  first.

### Checkpoint 4 — Bias + (optional) PCF + face culling
- **Compiles:** cull-mode set on the shadow encoder; epsilon slider;
  optional 2×2 or 4-tap PCF.
- **Renders:** Clean shadow (no acne on the cube's own lit faces, no
  detached peter-panning at the base), soft-ish edge if PCF is on.
- **Verify:** Cube's top/lit faces are uniformly lit (no shimmering).
  Shadow contacts the cube base cleanly. Bias sliders in the new panel
  produce visible, monotonic changes. Temporarily set the light to
  `{0,-1,0}` (straight overhead) to stress the degenerate-up path and
  grazing-angle acne.

### Checkpoint 5 — Scene-fit light frustum + polish
- **Compiles:** light ortho frustum computed from the **world-space
  AABB of all casters** (transform AABB corners into light-view space,
  take min/max → `ortho` L/R/B/T and near/far), recomputed each frame
  in `Application::onRender`. Editor panel finalized (toggle, bias,
  debug view).
- **Renders:** Shadows stay correct when objects are moved/added — no
  more "shadow disappears when I drag the cube outside a fixed box."
  Shadow resolution is well-used because the frustum hugs the scene.
- **Verify:** Drag the cube around / scale it; its shadow persists and
  stays reasonably crisp. Add a second caster; both cast and receive.
  Toggle shadows off → scene returns to the pre-shadow look exactly.
  Run the automation path (`--run-script` on a scene that just renders)
  to confirm the new pass doesn't break headless/automation rendering
  (it shouldn't, since rendering is outside the determinism contract,
  but worth a smoke check).

## Open questions (resolve before coding CP3+)

1. **Receiver:** the infinite `Ground` (grid) is the obvious "floor"
   but won't receive shadows without modifying `grid.metal`. For CP3,
   default to using a finite `Plane` as the receiver (smallest change).
   Alternative: extend the grid shader to sample the shadow map too
   (nicer but one more shader edit).
2. **Light frustum fit:** confirm the **scene-AABB fit** approach (CP5)
   rather than a fixed-size box. AABB-fit is ~15 lines and prevents a
   lot of "why did my shadow vanish" confusion, but it's the one piece
   that reads scene data outside the renderer.
